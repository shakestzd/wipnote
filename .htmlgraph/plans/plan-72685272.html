Plan: Auto-compliance after feature complete — OODA pilot

Pilot for the OODA active-observer initiative. When an agent runs `htmlgraph feature complete `, a PostToolUse hook fires `claude -p` (headless) in the background to compare the feature's spec against its implementation diff and write structured findings back to the work item HTML. Closes the Act → Observe edge of the OODA loop with low blast radius — single hook, scoped LLM call, structured output. Built on primitives confirmed in spike `spk-13b6115b`.

Problem

Today, completing a feature in HtmlGraph is a one-way write: the agent marks `feature complete`, the HTML body is updated, and the OODA loop's Act → Observe edge depends on a human later reading lineage and judging whether the implementation matched the spec. Specs grow stale; checkbox compliance is manually maintained or skipped; drift between spec and code is invisible until someone notices. Closing this edge with a hook-driven LLM judgment turns completion into a feedback event instead of a terminal one.

Goals

  1. **Automated spec-vs-implementation comparison** — every `feature complete` triggers an LLM judgment scored 0–100, written into a structured `<section class="compliance-findings">` on the feature HTML
  2. **Non-blocking UX** — the hook fires `claude -p` async; agent CLI completes within 100ms regardless of LLM latency
  3. **Cost-bounded** — per-call cost captured in `total_cost_usd` from the JSON result; `--max-budget-usd` ceiling per call; opt-in via `.htmlgraph/config.json` (default off)
  4. **Reusable harness** — the `internal/headless` package introduced here is the foundation for the other three OODA features (drift detection, orientation scoring, tempo metrics)

Constraints

  • No new runtime dependencies — Go stdlib + existing project deps only
  • Hook must respect the 'no stderr from hooks' rule — all `claude -p` stdout/stderr captured into buffers, never leaked
  • Plugin source-of-truth: any new hook wiring goes via `packages/plugin-core/manifest.json` and `htmlgraph plugin build-ports`, never direct edits to generated `plugin/hooks/hooks.json`
  • Default-off opt-in — pilot must be safe to merge to main without affecting users who haven't enabled it
  • Backward compatible — existing `htmlgraph compliance <id>` (checkbox reporter) behavior unchanged; new functionality is `htmlgraph compliance auto <id>`

#1 Headless invocation harness for `claude -p` Add internal/headless/claude.go — a Go package wrapping claude -p invocations. M Low Not Started Pending 2 Q
What

Add internal/headless/claude.go — a Go package wrapping claude -p invocations.

Public API:

  • type InvokeRequest struct { SystemPrompt, UserPrompt, Model, Effort string; AllowedTools []string; PermissionMode string; SessionID string; MaxBudgetUSD float64; Timeout time.Duration }
  • type InvokeResult struct { Result, SessionID, Subtype, Stderr string; IsError bool; TotalCostUSD float64; PermissionDenials []string; Usage map[string]any }
  • func Invoke(ctx context.Context, req InvokeRequest) (*InvokeResult, error) — runs claude -p --bare --output-format json --model <m> --effort <e> --max-budget-usd <n> --permission-mode dontAsk --allowedTools <list>, parses JSON
  • Package-level var Invoker InvokeFn = invokeReal for test stub injection (matching the existing completeIfInProgressFn pattern in internal/hooks/)

Captures stdout/stderr into bytes.Buffer so callers' stderr is never polluted. Honors ctx.Done() to kill the subprocess. Returns typed errors: BudgetExceededError (subtype error_max_budget_usd), ParseError, ExitError.

Why

Foundational primitive for all four OODA active-observer features (auto-compliance, drift detection, orientation scoring, tempo metrics). Encapsulating flag-building, JSON parsing, error handling, and stderr containment here means subsequent slices and follow-on plans don't reimplement them. Stub injection enables every downstream slice to be unit-testable without API spend.

Open Questions

Test stub injection — package-level var or interface? - **A: package-var** — `var Invoker InvokeFn = invokeReal`. Minimal change, easy to swap in tests, matches existing `completeIfInProgressFn` pattern in `internal/hooks/`. Test isolation depends on tearing down. - **B: interface** — Inject an `Invoker` interface via constructor. More idiomatic Go, but ~30 more lines and friction in callers. Recommended: **A — package-var**. Matches existing codebase pattern, faster to land, downstream slices need the stub yesterday.

What permission mode should the harness default to? - **A: dontAsk** — restrictive; tools must be in `--allowedTools` or call fails. Safest for unattended. - **B: bypassPermissions** — skips all checks. Simpler but accepts whatever the LLM tries. Recommended: **A — dontAsk**. Auto-compliance is unattended; dontAsk + explicit Read-only allow-list is the safest default for an OODA observer. Subsequent OODA features can override per-call.

What

Add cmd/htmlgraph/compliance_auto.go — a new CLI subcommand htmlgraph compliance auto <id> [--model claude-sonnet-4-6] [--effort medium] [--dry-run] [--max-diff-chars 50000].

Pipeline:

  1. Load feature node; error "no feature: <id>" if not found
  2. Read spec via extractSpecSection() (already exists in cmd/htmlgraph/compliance.go)
  3. If no spec section: write a "compliance skipped: no spec" finding via writeComplianceSection, exit 0
  4. Get commits attributed to the feature via db.GetCommitsByFeature
  5. Build diff blob: for each commit, run git show --no-color <hash>. Truncate at --max-diff-chars (default 50000) at the nearest line boundary; append ... [truncated N chars] footer
  6. Build prompt:
    • System: "Return ONLY valid JSON. No prose, no markdown fences. Schema: {summary: string, criteria: [{text, status: 'pass'|'fail'|'unclear', evidence}], score: 0-100, notes: string}"
    • User: spec + diff
  7. Call headless.Invoke from slice 1 with Model, Effort, AllowedTools: ["Read"], PermissionMode: "dontAsk", MaxBudgetUSD: 0.50, Timeout: 60s
  8. Parse result as JSON. If parse fails: write a "compliance error: parse failure" finding, exit 1
  9. Write a <section class="compliance-findings" data-score="N" data-cost-usd="0.012" data-model="claude-sonnet-4-6" data-timestamp="..."> with rendered findings. Idempotent: replaces existing section by class match, doesn't append.
  10. Print one-line summary to stdout: compliance feat-X score=N cost=$0.012
  11. --dry-run skips LLM call and section write — just prints the prompt that would be sent
Why

The pilot's value is testable manually as soon as this slice lands. Decoupling the pipeline from the hook trigger lets us dogfood compliance scoring on existing features before automation, validate the prompt quality empirically, and tune --model / --effort based on real outputs. Without this slice, the pilot only works through hook triggers — slow and expensive to iterate on.

Open Questions

Where should compliance findings live in the feature HTML? - **A: section** — `<section class="compliance-findings">` with `data-*` attributes. Structured, queryable, parallels existing `<section class="spec">`. - **B: addnote** — Use `Collection.AddNote` to append a `<p>` to body. Minimal new code, but findings buried in body and not idempotent. - **C: properties** — Store as `Properties` map on node. Simplest, but invisible in HTML view. Recommended: **A — section**. Idempotency via class match, queryability via existing extractor pattern, visibility on dashboard. AddNote loses replace-on-rerun semantics. Properties hides findings.

How does the LLM see the implementation diff? - **A: prompt-text** — pre-compute diff via `git show`, embed as text in user prompt; truncate at 50k chars - **B: bash-allowed** — allow `Bash(git show)` in `--allowedTools`, let LLM fetch diff itself - **C: read-files** — allow `Read` on changed files, let LLM read them Recommended: **A — prompt-text**. Deterministic, fully testable with stubs, avoids permission-mode complexity. Bash-allowed needs allow-list for git but not other commands — risky surface. Read-files doubles LLM calls (it has to figure out which files matter). 50k chars covers ~98% of features at our typical scope.

When the feature has no spec section, what should auto-compliance do? - **A: skip-with-note** — write a 'no spec' finding section, exit 0. Future runs can re-run after spec is added. - **B: silent-skip** — exit 0 with no side effect. - **C: generate-then-score** — run `htmlgraph spec generate`, score against the auto-generated stub. Recommended: **A — skip-with-note**. Visibility matters; silent skip leaves users wondering if the pilot is broken. The 'no spec' finding nudges users to add a spec. Generating a stub spec to score against is theater.

Default model + effort for auto-compliance? - **A: sonnet-medium** — `claude-sonnet-4-6` + medium effort. ~$0.01–$0.03/call. Good signal. - **B: haiku-low** — `claude-haiku-4-5` + low effort. ~$0.001–$0.003/call. May miss nuance. - **C: opus-high** — `claude-opus-4-7` + high effort. ~$0.10–$0.30/call. Overkill for pilot. Recommended: **A — sonnet-medium**. Pilot economics: 10 features/day = ~$0.30/day. Acceptable. Haiku struggles to ground claims in diff evidence. Opus is overkill at this scope. All three selectable via `--model` + `--effort`.

#3 PostToolUse hook trigger — fire `compliance auto` async on `feature complete` Wire the PostToolUse hook to detect htmlgraph (feature|bug|spike) complete and spawn htmlgraph compliance auto in the background. M Med Not Started Pending 3 Q
What

Wire the PostToolUse hook to detect htmlgraph (feature|bug|spike) complete <id> and spawn htmlgraph compliance auto <id> in the background.

  1. In internal/hooks/posttooluse.go, add a regex symmetric to existing featureStartRe: featureCompleteRe = regexp.MustCompile(\htmlgraph\s+(feature|bug|spike)\s+complete\s+(\S+)`)`
  2. After the existing featureStartRe match block, add a featureCompleteRe block that calls triggerAutoCompliance(projectDir, workitemID)
  3. New internal/hooks/auto_compliance.go with triggerAutoCompliance:
    • Read .htmlgraph/config.json for auto_compliance_enabled (default false)
    • If disabled: return immediately
    • If ANTHROPIC_API_KEY env var unset: return immediately (no stderr)
    • Build exec.Command(selfBinary(), "compliance", "auto", id)
    • cmd.Stdout = io.Discard, cmd.Stderr = io.Discard (no leak)
    • cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true} (detach process group)
    • cmd.Start() then cmd.Process.Release() (background, fire-and-forget)
    • Return immediately
Why

Closes the OODA Act → Observe edge automatically. Async (background spawn) keeps the agent's CLI fast — completing a feature is no longer blocked by claude -p latency. Default-off + API-key check keep the pilot safe to merge to main without surprising users. Reuses the established selfBinary() + exec.Command() pattern from task_tracking.go.

Open Questions

Synchronous (block hook on `claude -p`) or async (fire-and-forget)? - **A: async-default** — hook returns immediately, `claude -p` runs in background, findings appear when ready. UX unaffected by pilot. - **B: sync-with-timeout** — synchronous with 30s timeout; feedback before user sees completion output, but adds latency. Risk: hook timeout kills agent. - **C: configurable** — toggle in config.json; default async, opt-in sync. Recommended: **A — async-default**. OODA framing: active observers should not block the actor. Sync mode adds 10–30s to every feature complete. Findings landing 30s later in the dashboard is fine — they're for retrospective review.

Default value for `auto_compliance_enabled`? - **A: off** — default false; explicit opt-in. Pilot is safe to merge without surprising users. - **B: on** — default true; pilot is the new normal. Faster adoption. Recommended: **A — off**. Pilot phase: API costs are real, the prompt is unproven. Default-off lets us land the code in main and dogfood selectively. Flipping to default-on would be a separate slice / version bump after pilot success criteria are met.

What if the spawned `compliance auto` fails? - **A: silent** — failure invisible to the hook; findings simply absent from feature HTML. - **B: surface-on-next-cli** — write a 'compliance failed' finding section so the user sees it next time they view the feature. - **C: block-completion** — block the `feature complete` operation. (Defeats the OODA non-blocking principle.) Recommended: **B — surface-on-next-cli**. Visibility without blocking. Slice 2's command already writes failure findings on parse errors; same pattern for spawn-side failures. block-completion is incompatible with async.

#4 Cost telemetry, OTel signals, and compliance history Refinements layered on slices 1–3: M Low Not Started Pending 1 Q
What

Refinements layered on slices 1–3:

  1. Findings section attributes include data-cost-usd, data-model, data-tokens-in, data-tokens-out (read by dashboard for cohort cost reporting)
  2. Emit auto_compliance.complete OTel signal with feature_id, cost, duration, score — rolls up into existing otel_session_rollup
  3. Diff truncation: configurable max_diff_chars in .htmlgraph/config.json (default 50000); truncate at line boundary; findings include data-diff-truncated="true"
  4. Per-feature compliance history: a second invocation appends a <details> block within the section showing prior runs (last 5: timestamp + score + cost)
  5. New CLI: htmlgraph compliance history <id> — prints prior compliance runs as a table
Why

Pilot observability + safety. Without cost telemetry, we can't tell whether the feature is too expensive at scale. Without truncation, a large feature blows the context window and silently fails. Without history, regressions ('score dropped from 95 → 70 across runs') are invisible. These don't gate the pilot's value but make it sustainable past day 14.
