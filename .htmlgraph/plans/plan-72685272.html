Plan: Compliance auto-grading pilot — manual command + dogfood evaluation

Pilot for the OODA active-observer initiative. Adds `htmlgraph compliance auto ` — a manual CLI command that compares a feature's spec against its implementation diff via `claude -p` (headless) and writes structured findings to a `

` in the feature HTML. Pilot scope is intentionally manual: validates LLM judgment quality before automation locks in the prompt. Hook-driven automatic invocation and OTel telemetry are deferred to follow-on plans gated on pilot success. Built on primitives confirmed in spike `spk-13b6115b`. This rewrite reflects critique findings from opus (technical) and sonnet (scope) — most notably collapsing a premature harness abstraction, integrating concurrent-write safety up-front, and adding an explicit evaluation mechanism so the pilot can measure its own success.

Problem

Today, completing a feature in HtmlGraph is a one-way write: the agent marks `feature complete`, the HTML body is updated, and the OODA loop's Act → Observe edge depends on a human later reading lineage and judging whether the implementation matched the spec. Specs grow stale; checkbox compliance is manually maintained or skipped; drift between spec and code is invisible until someone notices. A manual `compliance auto` command introduces an LLM judgment step that can run on completed features, surfacing implementation-vs-spec drift while keeping the pilot's blast radius small enough to evaluate without committing to automation.

Goals

  1. **Manual compliance scoring** — `htmlgraph compliance auto <id>` runs `claude -p` headless, compares spec vs diff, writes 0–100 score + structured findings to a `<section class="compliance-findings">` in the feature HTML
  2. **Pilot-evaluable** — empirical cost measurement before approval; retrospective `--batch-since` mode runs against historical features; 14-day dogfood with manual quality audit on ≥10 findings produces a GREEN/YELLOW/RED promotion decision
  3. **Concurrent-safe by design** — atomic temp-file + rename in feature HTML write path; per-feature lockfile prevents lost updates; no silent corruption under concurrent invocation
  4. **Cost-bounded** — `--max-budget-usd` ceiling per call, `--max-turns N` tool-loop cap, `--max-wall-clock 5m` self-enforced timeout, `total_cost_usd` written into findings section for cohort cost tracking

Constraints

  • No new runtime dependencies — Go stdlib + existing project deps only; the `claude` binary is the one external dep, present in the development environment
  • PostToolUse handler is already registered in `packages/plugin-core/manifest.json`; this pilot does not modify the manifest, does not require `htmlgraph plugin build-ports` regeneration
  • Hook-driven automation explicitly out of scope; deferred to a follow-on plan after pilot evaluation passes (gates: see `q-pilot-success-bar`)
  • Backward compatible — existing `htmlgraph compliance <id>` (checkbox reporter) behavior unchanged; new functionality is `htmlgraph compliance auto <id>`
  • All `claude -p` stdout/stderr captured into buffers; never leaks to caller's stderr (forward-compat for the future hook plan's 'no stderr from hooks' rule)
  • All file writes to feature HTML use atomic temp+rename; existing `internal/workitem/htmlwriter.go` write path is upgraded to use atomic semantics for all writers, not just compliance

#1 `htmlgraph compliance auto <id>` — manual end-to-end run with safety integrated Add cmd/htmlgraph/complianceauto.go — a new CLI subcommand: L Med Not Started Pending 11 issues 4 Q
What

Add cmd/htmlgraph/compliance_auto.go — a new CLI subcommand:

htmlgraph compliance auto <id> [--model claude-sonnet-4-6] [--effort medium] [--dry-run] [--preview] [--max-diff-chars 50000] [--max-turns 5] [--max-wall-clock 5m] [--batch-since <date>]

Pipeline:

  1. Resolve project root via git -C <projectDir> rev-parse --show-toplevel. If not a git repo, write a "compliance skipped: no git history" finding section, exit 0.
  2. Acquire per-feature lockfile .htmlgraph/locks/compliance-<id>.lock with current PID. If lockfile exists with a live PID, exit 1 with "compliance already running for ". Defer cleanup on exit.
  3. Load feature node; error "no feature: " if not found.
  4. Read spec via extractSpecSection() (existing in cmd/htmlgraph/compliance.go). If no spec section: write a "compliance skipped: no spec" finding, exit 0.
  5. Compute spec hash (SHA-256 of spec content) for staleness tracking.
  6. Build diff blob: a. Try db.GetCommitsByFeature for attributed commits. b. If 0 rows: fall back to git -C <root> diff <branch-point>..HEAD -- <files> using db.ListFilesByFeature. c. If still empty: write "compliance skipped: no diff available" finding, exit 0. d. For attributed commits: git -C <root> show --no-color --first-parent <hash> (skip merge commit replayed history). Skip commits not present locally (shallow clone safety). e. Truncate at --max-diff-chars (default 50000) at nearest line boundary; append ... [truncated N chars from M commits] footer; set data-diff-truncated="true" on the section.
  7. Build prompt:
    • System: Return ONLY valid JSON. No prose, no markdown fences. Schema: {summary: string, criteria: [{text, status: 'pass'|'fail'|'unclear', evidence}], score: 0-100, notes: string}.
    • User: spec text + diff blob.
  8. Call inlined runHeadlessClaude(ctx, req) (in same package — no separate internal/headless harness; YAGNI per critique):
    • Args: claude -p --bare --output-format json --model <m> --effort <e> --max-budget-usd 0.50 --max-turns <n> --permission-mode dontAsk --allowedTools Read
    • Set HTMLGRAPH_AUTO_COMPLIANCE_RUNNING=1 in spawned env (forward-compat fork-bomb guard for the future hook plan)
    • Capture stdout into bytes.Buffer; capture stderr into a separate bytes.Buffer — never leak to caller stderr
    • Honor context cancellation
    • Self-enforce --max-wall-clock via time.AfterFunc killing the subprocess
    • Parse JSON; return typed BudgetExceededError for subtype: "error_max_budget_usd"
  9. Parse result field as JSON. If parse fails: write "compliance error: parse failure" finding, exit 1.
  10. Write <section class="compliance-findings" data-score="N" data-cost-usd="0.012" data-model="claude-sonnet-4-6" data-spec-hash="<sha>" data-timestamp="..." data-diff-truncated="false"> with rendered findings as HTML. Idempotent: replaces existing section by class match. Atomic: render full HTML to <id>.html.tmp.<pid>, fsync, os.Rename(tmp, path) (POSIX atomic rename).
  11. Print one-line summary: compliance feat-X score=N cost=$0.012 [stale-spec] — [stale-spec] appears if previous section's data-spec-hash differs from current.

Flag semantics:

  • --dry-run — skip the LLM call, print the prompt that would be sent
  • --preview — run the full pipeline (LLM call included) but print findings to stdout instead of writing to feature HTML
  • --batch-since <date> — iterate over all done features completed since <date>; rate-limit to 1 call per 30s to bound API spend

Atomic write upgrade in internal/workitem/htmlwriter.go:

  • Replace os.WriteFile(path, content, 0644) in WriteNodeHTML with: write to <path>.tmp.<pid>, f.Sync(), os.Rename(tmp, path) (POSIX atomic rename)
  • Add a featureWriteMu sync.Map (keyed by feature ID) for in-process serialization of WriteNodeHTML
  • This upgrade is a P0 safety addition; without it, slice 1's "idempotent replace" guarantee is a textbook lost-update race (per opus critique HIGH-2)
Why

The pilot's value is testable as soon as this slice ships: a human or agent runs compliance auto <id> on any completed feature, sees findings appear in the HTML, and judges the LLM's signal quality. Decoupling from a hook trigger lets us tune the prompt empirically without committing to automation that locks in a bad prompt. All safety guards (atomic write, fork guard env var, lockfile, truncation, turn cap, wall-clock cap) are integrated up-front because the manual pilot will run on real features in real concurrent contexts — these aren't refinements, they're correctness. The retrospective --batch-since mode unlocks evaluation against historical features, which slice 2's quality audit needs to bootstrap. The harness abstraction from the prior plan version was YAGNI per sonnet's critique; we inline runHeadlessClaude and extract a shared package only when a second OODA feature actually exists to consume it.

What

Pilot evaluation infrastructure — produces the evidence needed to make a GREEN/YELLOW/RED promotion decision for a follow-on hook-driven plan.

  1. Empirical cost measurement (one-time pre-merge task). Run htmlgraph compliance auto <real-feat-id> on one real completed feature with a non-trivial spec. Record the measured total_cost_usd from the JSON output. If materially different from the $0.03/call estimate, update q-model-default recommendation and re-baseline cost framing in q-pilot-success-bar.

  2. Retrospective batch baseline. Run htmlgraph compliance auto --batch-since 2026-01-01 (slice 1's batch mode) against historical features in this project. Output of that run becomes the day-0 evaluation dataset. Capture the run's aggregate cost.

  3. htmlgraph compliance history <id> command. Reads prior compliance runs from the feature's findings section's <details> blocks (last 5 retained by slice 1's idempotent replace, when slice 1's design is extended in a future iteration to preserve history) AND from the section's data-* attributes for the most recent run. Prints a table: timestamp, model, score, cost, spec-hash, diff-truncated. Supports --json for evaluation script consumption.

  4. Review checkpoint runbook — docs/runbook/compliance-pilot-evaluation.md describing:

    • Day 0 baseline: measured cost + retrospective batch dataset
    • Day 7 checkpoint: count of compliance auto invocations across dogfood users, sample of 5 random findings reviewed for plausibility, decision to continue or adjust prompt
    • Day 14 checkpoint: full quality audit — manual review of 10 findings against engineer judgment, score-agreement target ≥70%
    • Decision criteria: GREEN (all gates pass) → spawn hook-driven follow-on plan; YELLOW (signal noisy) → tune prompt, extend pilot 7 days; RED (consistent disagreement or unmanageable cost) → kill pilot, document why

  5. Dogfood activation. Project-level shell-alias or runbook entry directing dogfood users to run htmlgraph compliance auto <id> after each feature complete. Since the hook is out of scope, "activation" means establishing muscle memory and reporting cadence. The runbook explicitly states the 14-day window and the success criteria.

Why

The pilot has no value without an evaluation mechanism. Slice 1 produces findings; this slice produces evidence about whether those findings are useful. Without the empirical cost measurement, the model-default decision is built on extrapolation. Without the runbook, "pilot success" is a survey we never conduct. Without the history command, multi-run regressions are invisible. Without the retrospective batch, the pilot accumulates evaluation data only on features completed during the dogfood window — too small a sample. This slice converts a working tool (slice 1) into a measured experiment.
