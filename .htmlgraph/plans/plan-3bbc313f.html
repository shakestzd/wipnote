HtmlGraph Design Slices Decisions Critique Progress Chat

Plan: HTML-canonical architecture migration

Lock in a public HTML schema (JSON-LD payload + microdata body + link rel head edges + meta-name primitives + time datetime) that makes the work-item HTML files the canonical queryable substrate. Migrate every consumer (CLI, dashboard, ingest pipeline) off SQLite as a system requirement, in six phases. SQLite is allowed back later strictly as a disposable derived speed tier — not in this plan.

DesignPending

Problem

`.htmlgraph/htmlgraph.db` has corrupted three times in nine days. The architecture claims SQLite is a "derived read index" but the codebase treats it as authoritative: ~110 Go files import `database/sql`, the entire `internal/hooks/` package persists hook events directly to SQLite tables, `internal/db/` is a 18-file repo package, and core subsystems (`internal/graph/`, `internal/agent/`, `internal/planchat/`, `internal/registry/`) depend on SQL queries. Today the OTEL activity feed shows zero spans because the DB is empty/no-schema, even though 198 spans are durably written to NDJSON for the current session. The CLI read-path is ~85% HTML-direct, but the WRITE-path and dashboard are deeply SQLite-coupled. Driver: `modernc.org/sqlite v1.37.1`. The cost-to-benefit has flipped — SQLite imposes a corruption class on every session for speed wins that current scale doesn't meaningfully require, and every new feature accidentally adds another SQLite dependency.

Goals

  1. **Eliminate the corruption class** — losing the SQLite file becomes a sub-second nuisance (rm + rebuild from canonical HTML+NDJSON), not an incident.
  2. **Promote HTML to genuinely canonical** — every fact in a work-item file is queryable by an explicit, versioned schema (JSON-LD payload + `<link rel>` graph + `<meta name>` filters + `<time datetime>` + microdata body), informed by an audit of the actual data-* vocabulary in use today.
  3. **Make NDJSON canonical for events** — hook events and OTEL spans both persist to per-session NDJSON; SQLite is no longer the event store.
  4. **Remove SQLite as a system requirement** — the binary builds, ships, and runs without the `modernc.org/sqlite` driver; enforced via build tag (`!sqlite_disabled`) that swaps the import for an error-returning stub.
  5. **Preserve user-facing CLI and dashboard surface** — every existing command and dashboard view keeps behavior; users don't notice the migration.
  6. **Land a new `htmlgraph query` subcommand** — jq-style filter over JSON-LD payloads via embedded `itchyny/gojq`.

Constraints

  • No new runtime language deps (no Python, no Rust binary). Pure-Go embeds only.
  • Schema vocabulary versioned at `https://htmlgraph.dev/v1/`; freeze ONLY after corpus audit (slice-12) enumerates every distinct `data-*` attribute and edge type currently in use across the 577 existing HTML files.
  • Phase 0 (slices 1-2) must land first — we do not remove SQLite paths while the DB is the only thing serving the activity feed.
  • Every SQLite removal step keeps a non-SQLite fallback; the build-tag stub (slice-8 acceptance) is the binding compile-time contract.
  • Backfill of existing `.htmlgraph/*.html` files (slice-6) must be reversible (git-snapshot before write) and content-preserving (passthrough body field; post-render diff fails the migrator).
  • Plan blast radius extends well beyond the read-path audit: ~110 SQL-importing files, 18-file `internal/db/`, ~20-file `internal/hooks/` event-persistence layer, internal/{graph,agent,planchat,registry} all in scope. Several follow-on slices (12-16) added to cover work the original 11-slice plan missed.

#6 Phase 2b — Backfill migration of 577 existing `.htmlgraph/*.html` files L High Not Started Pending
What

Implement htmlgraph migrate-html --to v1: iterate every file under .htmlgraph/{features,bugs,spikes,plans,tracks,sessions}/*.html (577 files in the working repo). EXTRACT facts from the existing data-* attribute scheme that's actually on disk today (data-status, data-type, data-since, data-relationship, and others enumerated in the slice-12 v0 inventory). NOT from JSON-LD or microdata, which don't yet exist on disk.

Populate a schema.v1.<Type> struct including its BodyPassthrough field — capture every byte of body content not mapped to a known field, so hand-edits and freeform notes round-trip verbatim. Re-render via the new templater (slice-5) and write back.

Rollback: before the first write, the migrator runs git stash push -u on .htmlgraph/ (or commits a snapshot to a pre-v1-migration branch) so any failed migration can be reverted with one git command.

Modes: --dry-run prints semantic diffs and exits 0/1 based on diff cleanliness without writing. --write performs the rollback snapshot, then iterates with fail-loud on any file where the post-render whitespace-normalized body differs from the original. --check validates that every file under .htmlgraph/ already passes schema.Validate (used in CI to assert the migration converged).

Add a CI job that runs htmlgraph migrate-html --dry-run against the committed .htmlgraph/ snapshot in this repo (the repo dogfoods itself) and uploads the diff as an artifact for human review BEFORE the write path is enabled.

Why

Without backfill, the codebase has two HTML formats: schema-compliant (new emissions) and pre-schema (existing). The dual-format burden defeats the whole plan. Backfill collapses the world to a single format before Phase 3 reads can rely on it.

#7 Phase 3a — `internal/htmlparse` JSON-LD fast path + ParseLinks + ParseHeadMeta M Low Not Started Pending
What

Add three new exported functions to internal/htmlparse: (1) ParseJSONLD(r) (*schema.v1.<Type>, error) — extracts the first <script type="application/ld+json"> and unmarshals. (2) ParseLinks(r) ([]Link, error) — head-only stream parse for <link rel="..." href="..."> tags. (3) ParseHeadMeta(r) (map[string]string, error) — head-only stream parse for <meta name="htmlgraph:*" content="...">. All three short-circuit on </head>. Existing ParseFile continues to work via goquery for legacy callers and body-content extraction. Benchmark JSON-LD parse vs full parse and document the win.

Why

The fast paths are the perf justification for not needing SQLite for the common queries. ~1ms-per-file JSON-LD parse vs ~10ms goquery means a 200-item snapshot is ~200ms instead of 2 seconds — acceptable for CLI use, generous for dashboard cold-start.
