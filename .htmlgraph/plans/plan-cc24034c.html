Plan: CRISPI spec-definition enforcement — OpenSpec format + cross-harness elicitation

Closes the spec-definition gap in CRISPI: today, 0 of 117 done features have a `

` block. Adopts OpenSpec's MIT-licensed Requirement/Scenario template (RFC 2119 + Given/When/Then) as the canonical spec format. Adds a cross-harness elicitation command (works on Claude/Codex/Gemini) plus a Claude-convenience skill, both writing free-text decision notes onto the slice. Gates `promote-slice` and `feature complete` on spec presence (opt-in via config). This rewrite (v2) addresses 10 HIGH findings from the opus + sonnet critiques: deferred 3-dimension scoring (premature without data), dropped typed `Decisions` schema (YAGNI — replaced with free-text `DecisionsNotes`), fixed promote-slice gate temporal bug (gate the slice, not the not-yet-feature), added cross-harness CLI fallback for the skill, fixed concurrent-writer corruption via shared per-feature lock, made `--insert` non-destructive by default. Backfill of the 117 existing spec-less features is explicitly out of scope (see non-goals).

Problem

The CRISPI planning workflow (slice cards → promote-slice → feature complete) never enforces a spec block at any stage. Slice fields (what/why/done_when/tests) describe intent in plan YAML, but they don't reach the feature HTML where compliance tools look. Result: zero features have specs, the `htmlgraph compliance` checkbox reporter has nothing to read, and the auto-compliance pipeline shipped in feat-38a032e8 can only validate the skip-no-spec path. OODA active observation has no reference to compare implementation against.

Goals

  1. **Canonical spec format** — adopt OpenSpec's MIT-licensed Requirement/Scenario template as the structure of every `<section class="spec">` block, enabling structured parsing instead of free-text checkbox heuristics
  2. **Cross-harness elicitation by default** — every slice that promotes to a feature gets a spec via a simple 3-question elicitation (Scope/Decisions/Context). The CLI command works on every supported harness (Claude/Codex/Gemini); the Claude skill is a UX convenience layered on top.
  3. **Enforcement at two gates** — `promote-slice` cannot create a feature without slice-level decisions; `feature complete` cannot mark done without a non-empty spec block. Both opt-in via config toggle. Audited override available for emergencies.
  4. **Concurrent-writer safety** — extend the existing per-feature mutex from feat-38a032e8 so `spec generate --insert` and `compliance auto` cannot lose updates when run on the same feature

Constraints

  • OpenSpec template content is MIT-licensed; copy verbatim with attribution (HTML comment in generated specs + `THIRD-PARTY-NOTICES.md` at repo root, matching the Go ecosystem convention). DLAI SDD methodology has no license declared — do NOT copy verbatim from that repo; only the elicitation pattern is adapted.
  • Backward compatible — existing `[ ]/[x]` checkbox parser must keep working; new format is additive. Hybrid spec content (both formats in one section) is rejected by the parser via first-heading-match scoping rather than silently double-counted.
  • No new runtime dependencies — Go stdlib + existing project deps; OpenSpec's npm CLI is NOT a dependency.
  • Default-off opt-in — enforcement gates default to disabled in `.htmlgraph/config.json` so the plan is safe to merge without breaking existing workflows.
  • Cross-harness — the elicitation CLI must work on every supported harness. The Claude-only skill is a convenience wrapper, not the canonical entry point.
  • Concurrent-writer safety — `spec generate --insert` and `compliance auto` must serialise on a shared per-feature lock to prevent the lost-update race when both write different sections of the same feature HTML.
  • **Non-goal: 3-dimension Completeness/Correctness/Coherence scoring** — defer until the existing single-score `compliance auto` has produced data on ≥10 features that demonstrates the single score is insufficient. (Was original slice 4; sonnet HIGH critique flagged as premature.)
  • **Non-goal: backfill of the 117 existing spec-less features** — out of scope; tracked as a follow-on bug. The `q-cohort-strategy` plan question records the recommended deferral.
  • **Non-goal: spec versioning / change tracking / OpenSpec ADDED/MODIFIED/REMOVED delta semantics** — specs are append-only by convention until a follow-up plan addresses revisions. The Requirement/Scenario format is adopted; the delta-merge workflow is not.
  • **Non-goal: extractSpecSection HTML-aware parser upgrade** — known fragility (raw `strings.Index` truncates on nested `</section>`). Filed as a separate bug. Slice 1's `--insert` writes the section but does not exercise the read path; the existing fragility risk is not increased by this plan.
  • **Non-goal: project-level `spec_context` config injection** — was original slice 4. CLAUDE.md and AGENTS.md already provide project context to `claude -p` (since `--bare` was removed). A separate config field would duplicate without enforcing sync.

Implementation:

The pilot's compliance auto can read any spec content but can't write one. The htmlgraph spec generate stub today produces a free-text template that ends in stdout. Adding --insert makes a feature's spec real and parseable in one command. Adopting OpenSpec's structured Requirement/Scenario format gives compliance auto something machine-checkable. Refusing-by-default protects manual edits (opus HIGH critique caught the silent-clobber risk). The shared per-feature lock prevents the concurrent-writer corruption opus also flagged. This slice is intentionally generator-and-write-path-only; the parser extension lives in slice 2.

#2 Parser extension: OpenSpec Requirement format + `[F]` failed-status revival Extend parseCriteria in cmd/htmlgraph/compliance.go to recognise both the legacy [ ]/[x] checkbox format AND the new OpenSpec ### Requiremen… S Low Promoted Approved 3 issues 1 Q
What

Extend parseCriteria in cmd/htmlgraph/compliance.go to recognise both the legacy [ ]/[x] checkbox format AND the new OpenSpec ### Requirement: / #### Scenario: format. Format detection by first-heading match: ## Acceptance Criteria → legacy mode; ## ADDED Requirements (or ## MODIFIED Requirements) → new mode. Reject hybrid pollution: only checkboxes directly under ## Acceptance Criteria count as legacy criteria; only ### Requirement: headings directly under ## ADDED Requirements count as new. Revive the dead criterionFailed switch path by extending the regex to capture [F]/[f] as failed status.

Implementation:

  1. Update the criterionPattern regex from (?i)^[\s\-\d\.]*\[([x\s])\]\s+(.+)$ to (?i)^[\s\-\d\.]*\[([xfF\s])\]\s+(.+)$ and add the failed-status mapping: m[1] == "F" || m[1] == "f" → criterionFailed.
  2. Add a section-state machine to parseCriteria: track which top-level heading we're under (## Acceptance Criteria vs ## ADDED Requirements vs neither). Apply the format-specific parsing only inside its own scope.
  3. New format parsing: when inside ## ADDED Requirements or ## MODIFIED Requirements, treat each ### Requirement: <name> as one criterion. Status derivation per recommendation in q-criterion-status-mapping: look for - [x]/[ ]/[F] task lines inside any #### Scenario: block beneath the requirement; if all scenarios pass, status=passed; if any fails, status=failed; if any unchecked, status=unchecked.
  4. Update criterion struct and downstream consumers (computeCompliance, formatCompliance, JSON output) to handle criterionFailed as a real path — populate the previously-dead switch cases.
Why

Slice 1 generates the new format but the existing parser only recognises legacy checkboxes. Without this slice, htmlgraph compliance <feat-id> (the legacy reporter) reports zero criteria on every spec generated by slice 1 — useless. Extending the parser instead of rewriting it preserves backward compatibility for any spec written in the legacy format. First-heading-match scoping (per opus MEDIUM finding) prevents the parser from double-counting a meta-text mention of ### Requirement: inside a legacy ## Acceptance Criteria section. Reviving [F] is unrelated cleanup but cheap to bundle since the regex change is the same edit.

What

Add cross-harness elicitation, a free-text DecisionsNotes slice field (NOT a typed schema — sonnet HIGH critique flagged the original Decisions []SliceDecision as YAGNI), a Claude-only convenience skill, and two opt-in enforcement gates.

Implementation:

  1. Schema field. Add DecisionsNotes string (yaml decisions_notes,omitempty) to PlanSlice in internal/planyaml/schema.go. No typed sub-struct — free text only. No validator change required (string with omitempty validates trivially).

  2. CLI elicitation command. New cmd/htmlgraph/plan_elicit_decisions.go registers htmlgraph plan elicit-decisions <plan-id> <slice-num>. Accepts:

    • --scope <text>, --decisions <text>, --context <text> flags (programmatic / non-interactive)
    • OR stdin in YAML format with three keys (scope:, decisions:, context:) when no flags set and stdin is a tty (or --from-stdin flag in CI)
    • Combines the three answers into a single Markdown blob (e.g., ### Scope\n<text>\n\n### Decisions\n<text>\n\n### Context\n<text>) and writes to slice.DecisionsNotes
    • Atomic write to plan YAML
    • Cross-harness — works on Claude Code, Codex CLI, Gemini CLI without modification

  3. Claude convenience skill. New plugin/skills/spec-from-slice/SKILL.md with frontmatter name: htmlgraph:spec-from-slice, user_invocable: true. Body instructs Claude to: a. Read the slice card from the plan YAML b. Use AskUserQuestion to elicit Scope / Decisions / Context (3 grouped questions) c. Run htmlgraph plan elicit-decisions <plan-id> <slice-num> --scope "..." --decisions "..." --context "..." to write the notes d. Run htmlgraph spec generate <feat-id> --insert (slice 1's command) to materialise the spec e. Handle re-elicitation: if slice.DecisionsNotes already populated, ask whether to re-elicit, edit-in-place, or skip

  4. promote-slice gate. In cmd/htmlgraph/plan_promote_slice.go after the approval check (~line 87) and before the dep check (~line 97): if config.spec_enforcement.promote_slice is true and slice.DecisionsNotes == "": refuse with error message "slice <num> has no decisions; run htmlgraph plan elicit-decisions <plan-id> <num> first (or invoke /htmlgraph:spec-from-slice on Claude). Override with --allow-spec-skip if intentional.". Add --allow-spec-skip flag for audited override (logs the override into the slice's Comment field). Default behavior unchanged when config false.

    Note: this gates on the SLICE having decisions, NOT on the feature having a spec — opus HIGH flagged the original "feature spec" framing as a temporal mismatch (the feature doesn't exist yet at gate time). The slice's DecisionsNotes is the right gate target since slice 1's spec generator reads it.

In cmd/htmlgraph/workitem.go wiSetStatusWithAgent, when typeName == "feature" and status == "done": if config.spec_enforcement.feature_complete is true, read the feature HTML, call extractSpecSection, parse via slice 2's extended parseCriteria. Refuse if (a) section absent, OR (b) section contains 0 ### Requirement: headings AND 0 legacy [ ]/[x]/[F] lines. Threshold: spec section must have ≥1 ### Requirement: heading with non-empty SHALL line, OR ≥1 legacy line with non-empty criterion text. Add same --allow-spec-skip flag for override.

  6. Config fields. Extend internal/hooks/task_completion_gate.go (or wherever block_task_completion_on_quality_failure lives) with the new fields:

    "spec_enforcement": {
  "promote_slice": false,
  "feature_complete": false

Why

Without enforcement, slices 1+2 are tooling that nobody is required to use. The gates flip the default once config opts in. Cross-harness CLI is the canonical entry point; the skill is Claude-specific UX sugar.
