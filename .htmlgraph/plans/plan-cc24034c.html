Closes the spec-definition gap in CRISPI: today, 0 of 117 done features have a `

Adopts OpenSpec's MIT-licensed Requirement + Scenario template format (RFC 2119 + Given/When/Then) as the canonical spec block, and SDD's 3-question interview gate (Scope/Decisions/Context) as the elicitation pattern before slice promotion writes the spec. Problem

The CRISPI planning workflow (slice cards → promote-slice → feature complete) never enforces a spec block at any stage. Slice fields (what/why/done_when/tests) describe intent, but they live in plan YAML rather than in the feature HTML where compliance tools look. The result: zero features have specs, the existing `htmlgraph compliance` checkbox reporter has nothing to read, and the auto-compliance pipeline shipped in plan-72685272 can only validate the skip-no-spec path. Without specs, OODA active observation has no reference to compare implementation against.

Goals

  1. **Canonical spec format** — adopt OpenSpec's MIT-licensed Requirement/Scenario template as the structure of every `<section class="spec">` block, enabling structured parsing instead of free-text checkbox heuristics
  2. **Elicitation by default** — every slice that promotes to a feature gets a spec generated through a 3-question interview gate (Scope/Decisions/Context), eliminating the cold-start problem of empty specs
  3. **Enforcement at two gates** — `promote-slice` cannot create a feature without a spec; `feature complete` cannot mark done without a non-empty spec block. Both opt-in via config toggle for safe rollout.
  4. **Three-dimension verification** — `htmlgraph compliance auto` reports Completeness / Correctness / Coherence sub-scores (OpenSpec /opsx:verify-inspired) instead of a single 0-100 grade, with a project-level `spec_context` injected into the LLM prompt

Constraints

  • OpenSpec template content is MIT-licensed; copy verbatim with an attribution comment (e.g. `<!-- Spec template adapted from OpenSpec by Fission AI (MIT). -->`). DLAI SDD methodology has no license declared — adopt ideas (3-question interview gate) but do NOT copy any text verbatim
  • Backward compatible — existing `[ ]/[x]` checkbox parser must keep working for any spec written in the legacy format; new format is additive, not replacement
  • No new runtime dependencies — Go stdlib + existing project deps; OpenSpec's npm CLI is NOT a dependency
  • Default-off opt-in — enforcement gates default to disabled in `.htmlgraph/config.json` so the plan is safe to merge without breaking existing workflows
  • Plugin source-of-truth: any new skill goes via `packages/plugin-core/manifest.json` and `htmlgraph plugin build-ports` if cross-harness, or directly under `plugin/skills/` if Claude-only

What

Adopt OpenSpec's MIT-licensed spec template (RFC 2119 Requirement + Given/When/Then Scenario format) as the canonical structure of htmlgraph spec generate output. Add --insert flag that writes the generated spec directly into the feature HTML's <section class="spec"> block. Extend parseCriteria to consume the new Requirement/Scenario format alongside the legacy [ ]/[x] checkbox format.

Implementation:

  1. Copy /tmp/openspec/schemas/spec-driven/templates/spec.md content verbatim into a new file plugin/templates/spec-template.md with an attribution comment at the top: <!-- Spec template adapted from OpenSpec by Fission AI (MIT). https://github.com/Fission-AI/OpenSpec -->. Add an entry to a new top-level ATTRIBUTION.md documenting the third-party source + license.
  2. Update renderSpecMarkdown (currently renders # Spec/## Problem/## Acceptance Criteria checkboxes/## Files/## API Surface/## Notes) to render a hybrid: keep ## Problem, ## Files, ## Notes as CRISPI-specific context; replace ## Acceptance Criteria with ## ADDED Requirements containing one ### Requirement: <name> per slice criterion, each with at least one #### Scenario: block (WHEN/THEN). The slice fields populate as: done_when[i] → Requirement i (with the criterion text as the SHALL statement); slice's tests field → Scenario blocks under each Requirement.
  3. Add --insert boolean flag to specGenerateCmd. When true: locate the feature HTML, find or create <section class="spec">, write the rendered Markdown inside, save atomically via writeFileAtomicRaw. Idempotent: re-running replaces the section content (matches the pattern from compliance_auto.go's section writer).
  4. Extend parseCriteria in compliance.go to recognise BOTH formats: existing [ ]/[x] lines under ## Acceptance Criteria (legacy), AND ### Requirement: headings with their #### Scenario: children (new). For new format, criterion status is derived from a sibling - [x]/- [ ] task line inside the Scenario block (or unchecked by default).
  5. Fix the dead criterionFailed code path: extend the regex to capture [F] (or [f]) as failed status. Existing dead switch cases will become live.
Why

The pilot's existing compliance auto command can read any <section class="spec"> content but writes nothing into one. The htmlgraph spec generate stub today produces a free-text template that ends in stdout. Closing this gap means a single command (htmlgraph spec generate --insert <feat-id>) makes a feature's spec real and parseable. Adopting OpenSpec's structured Requirement/Scenario format gives compliance auto something machine-checkable rather than ad-hoc Markdown. Backward compat with checkboxes preserves any future legacy specs and keeps the dead [F] parser path from staying broken.

Open Questions

Where should the OpenSpec MIT attribution live?

The OpenSpec template content is copied verbatim into HtmlGraph. MIT requires attribution. Three placements differ in visibility and maintenance cost.

Extend `parseCriteria` to handle both formats, or write a new `parseSpec` and dispatch?

The legacy parser is tightly coupled to `## Acceptance Criteria` heading + `[ ]/[x]` regex. Adding the OpenSpec Requirement/Scenario format doubles the surface.

How is criterion status derived from the OpenSpec format?

What

Add a new user-invocable skill htmlgraph:spec-from-slice that runs the SDD-inspired 3-question interview before slice promotion writes the spec. Add a Decisions []SliceDecision field to the slice schema for storing the resolved choices.

Implementation:

  1. New SKILL.md at plugin/skills/spec-from-slice/SKILL.md with frontmatter name: htmlgraph:spec-from-slice, description: ..., user_invocable: true. Body instructs Claude to: a. Read the slice card from the plan YAML b. Use AskUserQuestion to elicit Scope (what's in/out for this slice), Decisions (key implementation choices with rationale), Context (tone/stack/patterns to follow) c. Save answers to the slice's Decisions field via htmlgraph plan edit-slice <plan-id> <num> --decisions-json <file> d. Run htmlgraph spec generate <feat-id> --insert (slice 1's command) to materialise the spec, with answers passed via --decisions-json flag (the generator reads the slice's Decisions field and pre-fills the OpenSpec template's Requirement context with them)
  2. Extend internal/planyaml/schema.go PlanSlice with Decisions []SliceDecision (yaml omitempty). New struct: type SliceDecision struct { ID, Title, Choice, Rationale string }.
  3. Extend validator: if Decisions is non-empty, each must have non-empty Title and Choice.
  4. Add htmlgraph plan edit-slice subcommand for writing decisions back into the YAML (or a focused --decisions-json <path> flag on an existing edit command if one exists; otherwise new).
  5. Extend htmlgraph spec generate to accept --decisions-json <path> flag that pre-populates the spec's Requirement context with the elicited decisions, weaving them into the rationale section per Requirement.
Why

Without an elicitation gate, slice promotion produces a spec from slice fields alone — sufficient for trivial slices but thin for slices with implementation tradeoffs. The 3-question interview is the minimum-viable elicitation pattern from SDD, transferred to CRISPI's slice-card model. Storing the resolved decisions in the slice schema preserves them for the generator (slice 1) AND makes them queryable for htmlgraph compliance auto (so the LLM grader can evaluate whether the implementation honored the documented decisions). The skill is user-invocable so it can be run manually before promote-slice; slice 3 wires it as an enforced gate.

What

Add two enforcement gates, both opt-in via .htmlgraph/config.json:

  1. promote-slice gate. In cmd/htmlgraph/plan_promote_slice.go promoteSliceFromYAML, after the approval check (line 87) and before the dep check (line 97): if config.spec_enforcement.promote_slice is true and the slice has no spec materialised (no associated feature with non-empty <section class="spec">), return a clear error: "slice <num> has no spec — run htmlgraph spec generate <feat-id> --insert first, or invoke /htmlgraph:spec-from-slice". If config is false (default), behavior unchanged.

  2. feature complete gate. In cmd/htmlgraph/workitem.go wiSetStatusWithAgent, when typeName == "feature" and status == "done", after id, err = resolveID(dir, id) and before col.Complete(id): if config.spec_enforcement.feature_complete is true, read the feature HTML, call extractSpecSection, fail if empty or contains no requirements/criteria. Error message: "feature <id> cannot be completed — spec section is empty or unscored. Run htmlgraph compliance auto <id> first."

  3. Add spec_enforcement config fields with defaults via the same pattern as block_task_completion_on_quality_failure (internal/hooks/task_completion_gate.go):

    "spec_enforcement": {
  "promote_slice": false,
  "feature_complete": false
}
Why

Without enforcement, slice 1 + slice 2 are opt-in tooling that nobody is forced to use. The gates flip the default: once enabled, the workflow can't bypass spec definition. Defaulting both gates to false preserves backward compat — the plan can merge to main without breaking existing CI or non-spec workflows. Consumers opt in by editing .htmlgraph/config.json. Clear error messages tell the user how to satisfy each gate. The gate at feature complete is the "back stop" — even if promote-slice is bypassed (e.g., manually creating a feature), completion still demands a spec.

#4 Three-dimension Completeness/Correctness/Coherence + spec_context Extend compliance auto to report three sub-scores (Completeness / Correctness / Coherence) instead of a single 0-100 grade, and inject proje… M Low Not Started Pending 2 Q
What

Extend compliance auto to report three sub-scores (Completeness / Correctness / Coherence) instead of a single 0-100 grade, and inject project-level spec-writing context into the LLM prompt.

Implementation:

  1. Update the JSON schema in compliance_auto's system prompt: replace score: 0-100 with dimensions: { completeness: 0-100, correctness: 0-100, coherence: 0-100 } and overall_score: 0-100 (computed from dimensions). Update the prompt instructions to define each dimension per OpenSpec's /opsx:verify:
    • Completeness: % of declared Requirements that have implementation in the diff
    • Correctness: % of Requirements where implementation matches the SHALL statement and Scenarios
    • Coherence: presence of design.decisions and whether implementation follows them; defaults to 100 if no decisions exist
  2. Update autoComplianceFinding struct and buildComplianceSystemPrompt to match the new schema.
  3. Update renderFindingsHTML (or equivalent) to display the dimensions table in the findings section: <table data-dimensions> with rows for each dimension + overall.
  4. Update findings section attributes: add data-completeness, data-correctness, data-coherence alongside existing data-score (which becomes the overall).
  5. Add spec_context field to .htmlgraph/config.json: 
    "spec_context": {
  "tech_stack": "Go 1.24, SQLite, plugin/static (HTML+JS), templated via gohtml",
  "rules": [
    "Use SHALL for requirements; avoid 'should' which is non-normative.",
    "Each Requirement gets ≥1 Scenario describing observable behavior.",
    "Decisions document WHY, not WHAT — the WHAT belongs in the Requirement."
  ]
}
  6. Read spec_context from config; inject into the system prompt of compliance auto's LLM call.
Why

A single 0-100 score collapses three different signals: did the implementation cover all requirements, did it implement them correctly, and did it stay coherent with documented decisions. The OpenSpec /opsx:verify pattern separates these for actionable feedback — a 30/100 in Completeness vs 30/100 in Coherence has very different remediation. The spec_context injection (analog to OpenSpec's config.yaml) gives the LLM project-specific rules so it grades against our conventions, not its priors.
