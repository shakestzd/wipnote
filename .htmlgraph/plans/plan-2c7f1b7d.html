Problem

Agent Teams plugin integration began in plan-14772c97 (April 2026) but was never formally promoted to features. Four of its six slices shipped to code informally and remain in production: the `EventTaskCreated` event-type constant, step-text teammate attribution in HTML, the opt-in quality gate on TaskCompleted, and the orchestrator skill's Agent Teams vs Subagents section. Two gaps remain. (1) **Persistence gap** — slice 1's DB write side: `AgentEvent` struct lacks `TeammateName`/`TeamName`/`IdleReason` fields, `InsertEvent` doesn't write them, the `idle_reason` column is missing from schema entirely (S1 specified `teammate_name`/`team_name` but not `idle_reason`), and `TeammateIdle` uses the shared `recordSimpleEvent` helper that doesn't propagate teammate fields. Result: dashboard queries that filter or aggregate by teammate return NULL on every row. (2) **Manual env-var setup** — users have to remember `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` per-shell or in `.claude/settings.json`, and the orchestrator skill still says "set it manually" because no auto-injection exists. Plus CLAUDE.md's brief monitoring note has accumulated drift, and v2.1.64+ JSON hook control payload (`{"continue": false, "stopReason": "..."}`) is undocumented.

Goals

  1. **Complete the persistence gap (was plan-14772c97 S1)** — `agent_events.teammate_name`, `team_name`, and `idle_reason` populated on every team-hook write. Closes the half-shipped state where parse-side captures the values but DB-side discards them.
  2. **Zero-friction enablement (net-new)** — `htmlgraph claude` / `htmlgraph yolo` / `htmlgraph dev` auto-inject `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`. Non-empty user-set env var (any value, including `0`) takes precedence; empty-string value treated as unset and overwritten.
  3. **Cost transparency** — first-run stderr notice + `.htmlgraph/AGENT_TEAMS_DEFAULT_v1` stamp file warns of Claude Code's ~7× token cost for teams. Future default flips don't break existing users.
  4. **Opt-out path** — `.htmlgraph/config.json` `agent_teams.enabled: false` disables auto-injection for hermetic isolation or debugging.
  5. **Accurate documentation (completes plan-14772c97 S6)** — CLAUDE.md and orchestrator skill flipped from 'set manually' to reflect auto-injection; document v2.1.64+ JSON control payload availability (informational — implementation deferred to a future plan); single-source via `docs/agent-teams.md` referenced from both files to prevent drift.

Constraints

  • Cross-harness scope: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` is a Claude-only env var. The Codex CLI and Gemini CLI launchers do NOT need the change — agent teams are a Claude Code feature only. Confirmed by research: codex.go and gemini.go don't share `buildClaudeLaunchEnv`.
  • Backward compat: existing slice/feature step prefix attribution via `addTaskStep` continues to work; this plan adds DB-side persistence without changing the HTML write path.
  • User-set env var precedence: `addIfUnset` already handles this correctly — only sets the env var if the user hasn't. No special logic needed.
  • Default-on opt-in: `agent_teams.enabled` defaults to true (auto-injection on). Opt-out is explicit `false` in config. Matches the user's stated intent ('we should natively set it because that improves orchestration').
  • **Non-goal: JSON control payload IMPLEMENTATION** — current `BlockExit2Error` + `os.Exit(2)` mechanism is correct on all Claude Code versions and works for the existing `block_task_completion_on_quality_failure` gate. Adding `{"continue": false, "stopReason": "..."}` JSON support requires refactoring `runHookNamed`'s blocking dispatch. Out of scope; only documented in CLAUDE.md as a v2.1.64+ feature available for future use.
  • **Non-goal: programmatic team creation** — Claude Code docs confirm teams are spawned via natural language only; no API exists. HtmlGraph's role is observation (hooks) and coordination (gates), not dispatch.
  • **Non-goal: backfill of historical agent_events rows** — pre-existing rows with NULL teammate_name/team_name remain NULL. New rows after this plan ships have the values populated. A separate plan can address backfill if needed.
  • **Non-goal: cross-harness teammate equivalents** — agent teams is Claude-only; no story for Codex/Gemini 'team' equivalents. Filed separately if those harnesses gain similar features.
  • All three slices must ship in the same release to prevent documentation drift — slice 3 documents what slices 1 and 2 implement; shipping them out of sync creates a window where CLAUDE.md says 'set the env var manually' while the launcher already auto-injects it (sonnet critic F1).
  • **Supersedes plan-14772c97** (finalized 2026-04-11). That plan's track `trk-6456fe7d` is orphaned (no canonical file in `.htmlgraph/tracks/`). The four shipped slices (S2 EventTaskCreated, S3 step-text attribution, S4 quality gate, S5 orchestrator skill section) are foundational and remain in production. This plan's slice-2 completes the half-shipped S1 (persistence). Slice-3 completes the partially-shipped S6 (CLAUDE.md docs). Slice-1 (auto-injection) is net-new — not in plan-14772c97.
  • **Track consolidation** — agent-teams work lives on `trk-8f9bc323` exclusively. The orphaned `trk-6456fe7d` reference in plan-14772c97 has no backing file and can be ignored. After this plan ships, plan-14772c97 is marked `completed` (its work either shipped to code or absorbed by this plan's slices).

Implementation:

    BlockOnQualityFailure bool `json:"block_task_completion_on_quality_failure"`

}
type agentTeamsConfig struct {
    Enabled *bool `json:"enabled"` // nil = default true; explicit false = opt out
}
    The *bool pointer is the established Go idiom for "absent means default". Add a helper isAgentTeamsEnabled(projectDir string) bool that returns true unless explicit false.
  4. In buildClaudeLaunchEnv, call isAgentTeamsEnabled(projectDir) before the addIfUnset call. Branch accordingly.
  5. Version-stamp first-run notice (per opus HIGH-4 cost-transparency finding): on the first launch where CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS would be auto-injected AND the stamp file .htmlgraph/AGENT_TEAMS_DEFAULT_v1 does not exist, emit a one-line stderr notice: "[htmlgraph] auto-enabling Claude Code agent teams (~7x token cost vs single agent). Opt out: agent_teams.enabled=false in .htmlgraph/config.json or CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=0 in env." then create the stamp file. Subsequent launches stay quiet. The stamp is per-project (lives in .htmlgraph/).
  6. Config struct location: launcher-local (per opus + sonnet medium critic findings). Define the new agentTeamsConfig struct in cmd/htmlgraph/claude_env.go itself (not in internal/hooks/task_completion_gate.go). Naming a generic htmlgraphConfig inside the hooks package couples unrelated concerns and a future reader looking for "the htmlgraph config" won't find it via package search. Launcher-local is correct for the current single consumer; promote to internal/config/ if a second consumer emerges.
  7. Empty-string env value precedence (per opus HIGH-1 finding): the existing addIfUnset helper treats empty-string user values as unset and overwrites them — its design is to rescue OTEL_* vars Claude Code zeroes out. The slice's done_when claim "any value takes precedence" must be tightened to "non-empty user-set value takes precedence; empty is treated as unset." If empty-string semantics matter to a user, document the workaround: CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=0 (explicit zero, non-empty).
  8. Config-vs-env precedence semantics (per opus medium): when agent_teams.enabled: false in config AND a non-empty CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS is set in the parent env, the planned implementation skips the auto-injection block entirely — meaning the env value passes through to claude unchanged. Document this in the slice's done_when AND in the stderr notice and CLAUDE.md: "agent_teams.enabled=false disables auto-injection only; an explicitly-set parent env var still passes through. To fully disable, both unset the env var AND set the config flag."
Why

The user's proposal: "we should natively set CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 because that improves the way htmlgraph orchestrates and manages the agents." Auto-injection eliminates the manual env-var setup step that today users have to remember per-shell or add to .claude/settings.json. The opt-out toggle provides a clean disable path for users who want hermetic isolation or are debugging team behavior. User-set env var precedence (already handled by addIfUnset) honors explicit user choice — including the value 0 to disable teams without touching config.

Without this fix, slice 1's auto-enablement would expand the surface of broken-by-omission behavior. Bundling the bug fix with the enablement plan ensures a coherent shipped behavior: when teams are auto-enabled, identity is also auto-persisted.
