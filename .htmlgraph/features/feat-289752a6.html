Description

The plan system v2 ships per-slice approve/reject/promote in the CLI (htmlgraph plan approve-slice / reject-slice / promote-slice / answer-slice-question / set-slice-status) and the data model supports it via plan_feedback section keys, but the dashboard side is half-built: GET /api/plans/ returns invalid JSON (parse error on the response), GET /api/plans//slices is HTTP 404, and there are no mutation endpoints at all. cmd/htmlgraph/serve.go registers only three plan routes (/api/plans, /plans/, /api/plans/) and the planRouter dispatch in cmd/htmlgraph/api_plans.go does not handle slice-level subpaths. Result: any plan with more than a handful of slices is unreviewable from the dashboard — the user has to drop to a shell for every slice action. This blocks the 16-slice master plan plan-3bbc313f from being usable as designed. Fix: wire up the missing read + mutation endpoints, reusing the same internal logic the CLI commands already call so there is one source of truth, and add the per-slice review UI in plugin/static/components.js. Verification target: open plan-3bbc313f in the dashboard, approve a slice, refresh, confirm approval persists.
