Description

Currently every OTel span stands alone in the trace view — you can't tell which work item the tool call belongs to. Hook-sourced events already carry feature_id (resolved via active_work_items table at hook time). OTel signals should inherit the same attribution so the span tree answers 'which bug/feature is this tool call advancing?'. Implementation plan: 1. Schema: add feature_id column to otel_signals (idempotent ALTER). 2. Receiver writer: on each INSERT, look up active_work_items.work_item_id for (session_id, agent_id=__root__) and populate feature_id. Cache within the batch so we don't query once per signal. 3. API: /api/otel/spans LEFT JOIN features ON features.id = otel_signals.feature_id so each span carries feature_id + feature_title. 4. Client: on rendered span rows, call this.featureBadge(span.feature_id, span.feature_title). Reuse the same badge styling as hook rows so trace-level attribution looks consistent with activity-tree attribution. Edge cases: subagents. For Task/Agent spans the work item may differ between parent and child (subagent may claim its own work item via feature start). active_work_items keys on agent_id too, so the right lookup is (session_id, agent_id) where agent_id = span's agent. But OTel doesn't carry agent_id — all signals in one session share session.id. This is an open question the Phase 5 plan should answer (span-based agent attribution). For now: use (session_id, __root__) and accept that subagent-level attribution will light up only after the orchestrator completes the subagent's work item. Minimum viable: top-level session attribution works. Subagent-level attribution is a follow-up.
