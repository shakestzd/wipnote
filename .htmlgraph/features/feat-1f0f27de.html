Description

Extend `parseCriteria` in `cmd/htmlgraph/compliance.go` to recognise both the legacy `[ ]/[x]` checkbox format AND the new OpenSpec `### Requirement:` / `#### Scenario:` format. Format detection by first-heading match: `## Acceptance Criteria` → legacy mode; `## ADDED Requirements` (or `## MODIFIED Requirements`) → new mode. Reject hybrid pollution: only checkboxes directly under `## Acceptance Criteria` count as legacy criteria; only `### Requirement:` headings directly under `## ADDED Requirements` count as new. Revive the dead `criterionFailed` switch path by extending the regex to capture `[F]`/`[f]` as failed status. **Implementation:** 1. Update the `criterionPattern` regex from `(?i)^[\s\-\d\.]*\[([x\s])\]\s+(.+)$` to `(?i)^[\s\-\d\.]*\[([xfF\s])\]\s+(.+)$` and add the failed-status mapping: `m[1] == "F" || m[1] == "f"` → `criterionFailed`. 2. Add a section-state machine to `parseCriteria`: track which top-level heading we're under (`## Acceptance Criteria` vs `## ADDED Requirements` vs neither). Apply the format-specific parsing only inside its own scope. 3. New format parsing: when inside `## ADDED Requirements` or `## MODIFIED Requirements`, treat each `### Requirement: ` as one criterion. Status derivation per recommendation in q-criterion-status-mapping: look for `- [x]/[ ]/[F]` task lines inside any `#### Scenario:` block beneath the requirement; if all scenarios pass, status=passed; if any fails, status=failed; if any unchecked, status=unchecked. 4. Update `criterion` struct and downstream consumers (`computeCompliance`, `formatCompliance`, JSON output) to handle `criterionFailed` as a real path — populate the previously-dead switch cases.
