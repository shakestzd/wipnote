Description

Symptom: htmlgraph CLI write commands (feature create, bug create, spike create, plan rewrite-yaml, etc.) intermittently fail with: open project: open database: applying pragmas: applying PRAGMA journal_mode: database is locked (5) (SQLITE_BUSY). Reproducibility: hit twice in a single planning session (2026-05-01 and 2026-05-03), both during sessions with a long-running htmlgraph claude --dev process active. Earlier writes in the same session succeed, then later writes fail, then succeed again on a later retry — pattern is intermittent, not persistent. Diagnosis evidence: fuser on htmlgraph.db showed the collector child running 'htmlgraph _serve-child --port 0' holding write locks. Read commands (status, snapshot --summary, plan show) work fine throughout — only writes that begin with 'applying pragmas: applying PRAGMA journal_mode' fail. Prior work: bug-62f14f8c reportedly addressed busy_timeout ordering (busy_timeout=5000ms applied atomically before journal_mode). Symptom persists, suggesting the fix is incomplete: either (a) 5s busy_timeout is insufficient under collector contention, (b) the timeout isn't being applied to the connection executing journal_mode, or (c) a different code path opens the DB without the pragmas guard. Impact: intermittently breaks scripted/automated htmlgraph CLI use. Will be a P0 blocker for the OODA active-observer pilot (plan-72685272) where a hook-driven background htmlgraph CLI write fires on every feature complete — at hook frequency the lock probability compounds. Reproduction: open htmlgraph claude --dev, do work for 30+ minutes, attempt write commands; observe intermittent failures. The lock clears within seconds-to-minutes, so retries eventually succeed.
