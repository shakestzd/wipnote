Description

Mirror of GitHub issue #89 (https://github.com/shakestzd/htmlgraph/issues/89). The YOLO-mode pre-commit visual-validation gate (yolo_guard.go's checkYoloUIValidationGuard) blocks commits with 'UI files were modified but no visual validation was performed' even when screenshots were taken inside mcp__claude-in-chrome__browser_batch calls. Root cause: the SQL predicate requires tool_name='mcp__claude-in-chrome__computer' AND tool_input LIKE '%"action":"screenshot"%' (with screenshot LIKE catch-alls). When a screenshot is taken via mcp__claude-in-chrome__browser_batch with actions:[{name:'computer',input:{action:'screenshot',tabId:N}}], the recorded tool_name is browser_batch, NOT computer. tool_input still contains 'action:screenshot' but the AND-predicate on tool_name excludes the row. Result: the gate fights the recommended Claude Code workflow (system reminders explicitly recommend browser_batch for MCP call batching). Fix (one-line SQL change in yolo_guard.go's checkYoloUIValidationGuard): - Drop the tool_name AND-filter, match on tool_input alone (any tool whose input contains action:screenshot) - Plus existing tool_name LIKE catch-alls for screenshot/take_screenshot Or more conservative: add OR (tool_name='mcp__claude-in-chrome__browser_batch' AND tool_input LIKE '%action:screenshot%'). Workaround until fixed: call mcp__claude-in-chrome__computer with action:screenshot directly, bypassing browser_batch.
