Description

Surfaced by htmlgraph migrate-tracks --dry-run after feat-d9c66e16 shipped (PR #73). Of 113 classified features, 85 returned 'no-attribution (0 files)' — meaning they have zero rows in the feature_files table. This caps the entire lineage system: htmlgraph blame returns false negatives on real touches (e.g. internal/graph/algorithms.go shows zero features even though git history attributes feat-918f08b9 and feat-27e4ecf1 to it); htmlgraph code-areas under-counts by 75 percent; htmlgraph migrate-tracks cannot re-attribute orphan features. The four value-aligned tracks (Subagents, Yolo Mode, Lineage, Plan Mode) will look perpetually under-populated until this is fixed. Likely root cause: features predate the feature_files indexing pipeline OR the indexer only captures attribution from active hook events, missing commits referenced via trailer / commit message conventions. Proposed remediation: add a 'htmlgraph reindex --backfill-orphans' mode that walks git log for every feature ID in the store, parses commit messages and trailers for feat-XXX references, extracts touched paths via git show --name-only, and populates feature_files retroactively. Smoke test post-fix: re-run htmlgraph migrate-tracks --dry-run and verify the no-attribution count drops below 20. Out of scope for this bug ticket: implementing the backfill is itself feature-sized work. This bug exists to surface the gap and track its resolution.
