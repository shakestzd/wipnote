Description

All 3 collector spawn retries hit a 3s handshake timeout and the launcher exits FATAL, so claude --dev never starts. Repro: htmlgraph claude --dev Observed output: Removing marketplace htmlgraph plugin for dev mode... Marketplace htmlgraph removed (uninstalled, disabled, cache wiped). Launching Claude Code with local plugin (--plugin-dir mode) htmlgraph: warning: collector spawn attempt 1/3 failed: otel-collect: handshake timeout (3s) htmlgraph: warning: collector spawn attempt 2/3 failed: otel-collect: handshake timeout (3s) htmlgraph: FATAL: collector spawn failed after 3 attempts: otel-collect: handshake timeout (3s) Likely root cause (untriaged): Stale claude --dev sessions and their otel-collect children survive across user disconnects in Codespaces. ps -ef shows two prior 'htmlgraph claude --dev' processes still alive (one from Apr27, one from Apr30) and at least one orphaned 'htmlgraph otel-collect' child holding resources. A freshly spawned collector then races with these for the SQLite project DB lock and cannot complete its pragma-application phase within the 3s handshake budget. Concurrent CLI calls in the same shell confirm SQLITE_BUSY: 'database is locked (5)' on PRAGMA journal_mode. Hypotheses to probe (in order): 1. Stale-collector reaping: detect zombie collectors at launch (pid file, port probe, or socket lockfile) and reap before spawning the new one. 2. Handshake budget too tight when another writer holds the DB. 3s is fine for a cold open, not for contention; either retry the handshake itself or extend the budget under detected contention. 3. --tmux mode (recommended in CLAUDE.md for Codespaces) is opt-in; non-tmux launches get killed silently on disconnect, leaving orphans. Consider making tmux the default for --dev in Codespaces, OR add a 'graceful-shutdown-on-parent-disconnect' supervisor that reaps the otel-collect when the parent claude exits. Severity: P1 — blocks dev-mode entry entirely; user cannot iterate on the plugin without manual ps/kill cleanup. Then rm the per-user cache DB at ~/.cache/htmlgraph//htmlgraph.db and retry.
