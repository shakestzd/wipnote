Description

Regression test added in baf16572 to guard the busy_timeout-before-journal_mode fix passes locally on the dev container but fails on GitHub Actions CI with: 'applying PRAGMA journal_mode: database is locked (5) (SQLITE_BUSY)'. Three failed CI runs since the fix landed: 25260804125, 25262577080, 25264218716. All on main. Working hypothesis (unconfirmed): the explicit 'PRAGMA busy_timeout=5000' exec at the start of ApplyPragmas runs on whatever connection database/sql hands out from the pool. The subsequent 'PRAGMA journal_mode=WAL' may run on a DIFFERENT pooled connection where busy_timeout has not yet been applied — making the timeout effectively zero on that connection. The DSN-level '?_pragma=busy_timeout(5000)' should cover this on every new connection, but if the modernc.org/sqlite driver doesn't apply DSN pragmas before the first explicit Exec, the race window opens. Production code may still be sound (manual stress repro showed 0/20 failures), but the regression test is exposing the fragility. Fix space: 1. Pin all pragma applications to a single dedicated *sql.Conn (db.Conn(ctx) + Conn.ExecContext). Forces serial execution on one underlying connection. 2. Set sql.DB.MaxOpenConns(1) before ApplyPragmas, restore after. Less invasive but ugly. 3. Confirm the modernc.org DSN pragma syntax actually applies busy_timeout on connection establishment (read driver source / docs). If not, the DSN is dead weight and the explicit exec must be on a pinned conn. Either way, the test must pass deterministically on CI. After fix, stress run go test ./internal/db/ -count=20 locally to confirm.
