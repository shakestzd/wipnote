Plan: Local cockpit — parallel AI sessions in the dashboard

The local cockpit extends feat-52d15f78 (the ttyd embed MVP that shipped a single hardcoded `htmlgraph claude --dev` iframe) into a full multi-pane session manager embedded in the HtmlGraph dashboard. Users can spawn N concurrent PTY sessions — Claude, Codex, Gemini, or yolo — each scoped to its own CWD or worktree and tied to a specific work item, all from a launcher modal without leaving the browser. The backend remains a thin generic PTY spawner: one endpoint accepts {agent, mode, cwd, work_item} and computes the shell command; no agent-specific logic leaks into internal/terminal. The frontend, staying vanilla JS, owns all launcher UX, pane layout, and inventory reload. Because every spawned session writes its own session_id + project_dir row through the existing observability pipe, the lineage graph gains a natural record of which pane produced which artifacts — making parallel multi-agent work traceable at the session level.

DesignPending

Problem

The MVP shipped in feat-52d15f78 puts one iframe in the dashboard that hardcodes `htmlgraph claude --dev` as the shell command. There is no way to open a second session, no way to pick Codex or Gemini as the agent, no way to scope a pane to a feature worktree, and no way to recover live panes after reloading the dashboard tab. The backend endpoint POST /api/terminal/start only accepts {work_item} and closes over the server-start projectDir — meaning all sessions run in the same directory regardless of what the user wants. A user running three parallel yolo agents on different features must leave the dashboard and manage tmux sessions manually, losing the observability lineage that HtmlGraph is designed to provide.

Goals

  1. **N concurrent panes** — dashboard can host ≥4 simultaneous live PTY sessions without collision
  2. **Agent-agnostic spawn** — same backend endpoint launches claude, codex, gemini, or htmlgraph yolo by parameter, not by new route
  3. **Per-pane scope** — each pane can run in the main repo, a feature worktree, or a track worktree with HTMLGRAPH_PROJECT_DIR routed correctly
  4. **Persistent pane inventory** — reloading the dashboard restores visible panes; ttyd processes outlive the browser tab
  5. **Work-item attribution preserved** — every spawned session writes distinct session_id + project_dir rows, visible in the lineage graph

Constraints

  • Backend stays generic PTY spawner — no agent-specific logic in internal/terminal
  • Frontend stays vanilla JS — no new framework/bundler introduced for cockpit alone
  • Dev mode launchers must use --plugin-dir via removeMarketplaceHtmlgraph() (already enforced) — cockpit must not drop that guard
  • bug-6ab963d3 and bug-d2d3fb3f are resolved via the checkServeLock PID-file pattern in claude_serve_autostart.go:173–215; do NOT add duplicate serialization at the terminal-handler layer.
  • feat-af47836b (logical session splitting) must land before multi-pane attribution is clean; cockpit ships alongside but leans on it for correctness
  • Cockpit endpoints are localhost-bound; any remote-exposure story is deferred (ttyd --credential is available when we need it)
  • Multi-project / global-serve mode (parent/child architecture in serve.go + serve_parent.go) is out of scope for v1; cockpit targets single-project mode only. Verify after landing that terminal routes propagate correctly through buildSingleProjectMux on the child side.
  • Backend session-reaper goroutine in internal/terminal/terminal.go:108–113 removes entries on ttyd exit; frontend must have a liveness signal (see q-liveness-signal) to avoid dead-iframe UX.

#1 Generalize Manager.Start request surface M Med Pending
What

All subsequent cockpit slices depend on the ability to spawn different agent types with different CWDs from a single endpoint. Without this generalization the frontend launcher modal has nowhere to send its parameters and every new agent would require a new route.

#2 UUID-keyed sessions + inventory endpoints M Low Pending
What
Add an id string (UUID v4) field to the session struct and change the in-memory map key from pid (int) to UUID. Add GET /api/terminal/sessions returning a JSON array of {id, agent, mode, cwd, work_item, port, started_at, state} for all live sessions. The state field uses values pending | live | exited. Add POST /api/terminal/stop accepting {id} (preferred) or {pid} (back-compat, drop in a later cleanup slice) to kill a session. Add POST /api/terminal/stop-all that kills all live sessions; used by the beforeunload beacon in slice-5 (navigator.sendBeacon body is small, fits the stop-all pattern cleanly). started_at is set at spawn time (time.Now()). UUID generation uses crypto/rand — no new dependency.
Why
The floating-window frontend needs a stable, reload-safe identifier per pane that is not a PID (which can be reused by the OS after a process exits). The inventory endpoint lets the dashboard reconstruct its pane list on reload by fetching live sessions and rendering iframes pointing at the live ttyd ports. The state field enables the async spawn model recommended in q-spawn-sync-model.

#5 Multi-pane shell (floating windows) L Med Pending
What
Extend the single-overlay pattern to support N simultaneous floating pane windows. Each pane is an independent div.terminal-pane containing a titlebar (agent label, work item, close button) and an iframe pointing at the live ttyd port. Panes are draggable via the titlebar (mouse events, no library). On dashboard load, call GET /api/terminal/sessions and re-render a pane for each live session (iframe src = http://127.0.0.1:<port>). Closing a pane calls POST /api/terminal/stop and removes the element. A global pane registry (Map keyed by session UUID) tracks open panes and prevents duplicate iframes for the same session. Tile/split layout is explicitly deferred to a future slice — floating windows only for v1. Lifecycle additions: - beforeunload event fires navigator.sendBeacon to POST /api/terminal/stop-all (defined in slice-2) — cleaner than iterating individual PIDs and stays within beacon body limits. - Pane count cap: the launcher disables the Create button when ≥6 panes are live; warning text explains the cap. - Pane position persistence is NOT implemented; reload cascades new panes from top-left (scope discipline — kick persistence to a future slice if needed). - When a ttyd process exits (user Ctrl-C inside the pane), the pane shows an "Ended" state within 5s. Mechanism governed by q-liveness-signal (poll /api/terminal/sessions every N seconds vs SSE).
Why
This is the visible cockpit: without multi-pane the whole plan is still a single-session wrapper. Floating windows are chosen over tiling because they require no split-pane JS library and each pane's position is independently user-controlled. The reload-from-inventory behavior makes the cockpit resilient to browser events without requiring server state to track frontend layout.

Alternative Approaches

  • Alt Single shared ttyd with multiplexed sessions considered and rejected: ttyd does not natively multiplex; would require tmux or screen as the inner shell, adding complexity.
