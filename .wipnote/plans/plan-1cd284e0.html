HtmlGraph Design Slices Decisions Critique Progress Chat

Plan: Per-session OTel collector with NDJSON ingest + SQLite as derived index

Move the OTLP receiver out of htmlgraph serve into per-session collector subprocesses that append OTel signals to session-scoped NDJSON files, and build an NDJSON-to-SQLite indexer in serve to preserve existing dashboard APIs. Retires the shared-port shared-writer model that causes parallel-session merging (bug-e8b8900b), SQLITE_BUSY cascades on the OTel write path (bug-08681166 class), and serve-restart span drops. Establishes the NDJSON-canonical + SQLite-derived pattern without committing to full SQLite removal (deferred to follow-on plans).

DesignPending

Problem

The OTLP receiver is embedded inside htmlgraph serve and writes to the shared .htmlgraph/htmlgraph.db SQLite file on a project-hashed port (4318 + FNV32a(projectDir) % 1000, internal/otel/receiver/port.go:11). This creates four recurring failure modes. (1) Parallel htmlgraph claude terminals in the same project share one port and one writer; Claude Code's span tagging is the only thing distinguishing their sessions, and bug-e8b8900b documents session merging from this. (2) The receiver competes with hook processes for SQLite writes, producing SQLITE_BUSY cascades — bug-08681166 (in-progress), and the already-resolved bug-6ab963d3 and bug-59b98285, all describe the same contention class. (3) Restarting serve drops in-flight spans because Claude Code's OTel SDK has no durable retry against a closed port (claude_serve_autostart.go:93 logs a warning and accepts the loss). (4) Session identity is authoritative only in Claude Code — HTMLGRAPH_SESSION_ID leaks from parent shells into subprocesses (internal/hooks/runner.go:187), misattributing hook events in subagent contexts.

Goals

  1. **Per-session isolation** — each htmlgraph claude/yolo/dev session owns its own collector subprocess, ephemeral port, and NDJSON file; parallel terminals never contend at the ingest layer.
  2. **Eliminate OTel writer collisions with SQLite** — collectors never hold a SQLite write lock directly; the bug-08681166 class of failures stops reaching the OTel ingest path entirely.
  3. **Lifecycle independence from serve** — collectors survive serve restarts without dropping spans; serve becomes a pure reader + semantic-op server.
  4. **Launcher-owned session identity** — htmlgraph mints the session ULID and binds it to every emitted event, replacing Claude Code's span-tagged session.id as the authority.
  5. **Zero dashboard regression** — every /api/otel/* endpoint keeps returning the same data shape; the NDJSON→SQLite indexer preserves all existing query paths.

Constraints

  • Keep SQLite as the dashboard read index for this plan. Removing it entirely is a future phase that depends on answers we do not yet have for six genuine-SQL sites: yolo-guard LIKE scans (yolo_guard.go:539-704), DSL graph traversal (internal/graph/dsl.go), feature_files self-join overlap (feature_files_repo.go:86), claims CAS and lease expiry (claim_repo.go), OTel span re-parenting transactions (writer.go:553), and dashboard sessions list 5-table correlated subqueries (api.go:116-146).
  • Reuse the existing syscall.Flock pattern (session_html.go:147, materialize.go:241) for NDJSON atomicity on oversize writes. Do not introduce a new lock primitive.
  • PIPE_BUF atomicity is 4096 bytes on Linux and 512 bytes on macOS. Some transcript_assistant.go:267 payloads exceed 4KB. Plan must name an explicit size-overflow strategy, not assume single-syscall atomicity.
  • Serve auto-spawn must continue to work for semantic ops and the dashboard — only the OTLP receive-side is removed from serve. The existing checkServeLock/writeServeLock/spawnDetachedServe pattern (claude_serve_autostart.go:179-255) is reused for the dashboard path.
  • Dashboard is an SPA (plugin/static/dashboard/js/app.js, 3435 lines) with substantial client-side aggregation already. The indexer preserves current API response shapes; no frontend rewrite is in scope.
  • Launcher currently detaches subprocesses via Setpgid:true and uses PID lockfiles. Collector subprocess reuses these mechanisms. Linux-only PDEATHSIG is not required if SIGTERM-on-exit is wired reliably at layered levels (SessionEnd hook + deferred launcher SIGTERM + idle timeout).
  • **Platform scope: Linux and macOS only.** Setpgid:true, PID-lockfile semantics, and file-locking patterns used throughout this plan are POSIX-specific. Windows support is explicitly deferred to a separate future plan; existing htmlgraph Windows coverage (if any) is neither extended nor regressed.
  • **fsnotify is not currently a dependency** (verified 2026-04-24 against go.mod). Q2 recommendation of fsnotify-with-polling-fallback adds it; the polling-only alternative avoids the new dependency. Either path is acceptable — the choice is a pure latency/dependency tradeoff with no other architectural consequence.
  • **Retention archives use stdlib archive/tar + compress/gzip** (.tar.gz), not .tar.zst. zstd is not currently a dependency and stdlib gzip is adequate for append-only event logs; this avoids introducing a new compression dependency for slice 7.
  • **The SignalSink abstraction is asymmetric by design.** The SQLite sink carries all placeholder/upgrade span re-parenting logic (writer.go:300-336) because that logic requires cross-table DB queries. The NDJSON sink appends raw signals only; the indexer (slice 5) re-runs the upgrade logic when replaying NDJSON into SQLite. Slice 1 must treat the two sinks as implementing the same interface but with intentionally different capabilities beyond raw append.

#1 Extract SignalSink interface; add NDJSONSink implementation M Low Pending
What
Refactor internal/otel/receiver/writer.go to depend on a SignalSink interface. The interface method signature matches the existing Writer's WriteBatch exactly: WriteBatch(ctx context.Context, harness otel.Harness, resourceAttrs map[string]any, signals []otel.UnifiedSignal) error. Keep the current SQLite-backed writer as the first concrete implementation under internal/otel/sink/sqlite/, preserving its placeholder/upgrade span re-parenting logic (writer.go:300-336) intact — that logic requires cross-table DB queries and stays with the SQLite implementation. Add internal/otel/sink/ndjson/ that opens .htmlgraph/sessions/<sid>/events.ndjson (layout per Q1) in O_APPEND mode and serializes each UnifiedSignal as one JSON line. NDJSONSink does NOT implement placeholder/upgrade — it is append-only over raw signals; the indexer (slice 5) re-runs upgrade logic when replaying. Oversize payloads are handled per the strategy chosen in Q4. Both sinks are constructable behind the same interface; internal/otel/receiver/receiver.go's New() accepts a SignalSink rather than opening a *sql.DB directly. The existing http.go:162 call site is updated to pass the selected sink.
Why
Today writer.go is deeply SQLite-coupled: five prepared statements, BEGIN IMMEDIATE transactions, cross-table FK resolution against pending_subagent_starts and active_work_items. Without an interface boundary, a per-session collector cannot choose between sinks and cannot be tested without a real SQLite file. This interface extraction is the prerequisite that unblocks every downstream slice. Supports goal Eliminate OTel writer collisions by making sink selection explicit. The asymmetric-capability design (SQLite sink carries upgrade logic, NDJSON sink does not) is load-bearing: without it, the indexer in slice 5 would have no place to re-run span re-parenting during replay.

#5 NDJSON to SQLite indexer loop inside serve L Med Pending
What
New internal/otel/indexer/ package that watches .htmlgraph/sessions/*/events.ndjson files (per Q2), tails each file, parses one JSON line at a time, and applies it to SQLite through the SQLite SignalSink from slice 1. The indexer re-runs the placeholder/ upgrade span re-parenting logic (relocated from writer.go:300-336 into internal/otel/indexer/upgrade.go) when applying NDJSON batches — NDJSONSink has no DB access, so the upgrade pass must happen here, during apply-to-SQLite, not at sink time. Maintains a per-file checkpoint in .htmlgraph/sessions/<sid>/.index-offset (written atomically via write-to-temp-then-rename after each successful batch commit; on corrupted/truncated checkpoint read, falls back to offset 0 and replays the whole file — INSERT OR IGNORE makes this idempotent). On indexer boot, replays from last checkpoint to end-of-file for every session directory. Mount the indexer in cmd/htmlgraph/serve_child.go alongside the existing autoIngestLoop; feature-flagged via HTMLGRAPH_INDEXER=1 if Q5 selects parallel-validation rollout, or unconditionally on if Q5 selects cutover.
Why
With collectors writing NDJSON instead of SQLite directly, every /api/otel/* endpoint (backed by otel_signals, otel_resource_attrs, otel_session_rollup) would break on day one. The indexer is the bridge: NDJSON is canonical, SQLite becomes derived and rebuildable. Relocating the upgrade/placeholder logic here is mandatory — it requires cross-table DB queries (pending_subagent_starts, etc.) that cannot live in the append-only NDJSON sink. Checkpoint files are what make rebuild by deleting htmlgraph.db and restarting serve a single-command recovery. Checkpoint truncate-to-zero fallback is safe because INSERT OR IGNORE on otel_signals.signal_id (verified writer.go:71) guarantees duplicate inserts never overwrite data. Directly supports goal Zero dashboard regression.

#7 Session cleanup: graceful drain + retention hygiene M Low Pending
What
Two pieces. (1) On SessionEnd hook, read .htmlgraph/sessions/<sid>/.collector-pid (written by slice 3), send SIGTERM, wait up to 3s for clean drain, fall back to SIGKILL if needed. This is Q3 primary layer; the collector own idle-timeout (slice 2) is the tertiary safety net; the launcher deferred SIGTERM (slice 3) is the secondary fallback. (2) Retention job in a new internal/otel/retention/ package: on serve startup and every 24h thereafter, walk session directories, detect sessions whose sessions.status = done is older than HTMLGRAPH_SESSION_RETAIN_DAYS (default 30), archive events.ndjson into .htmlgraph/archive/<yyyy-mm>/<sid>.tar.gz using stdlib archive/tar + compress/gzip (not .tar.zst — zstd is not a current dependency), and delete the live session dir (indexer-populated SQLite rows age out via existing cleanup commands, unchanged). Dry-run mode via HTMLGRAPH_RETENTION_DRYRUN=1 logs intended actions without moving files. Also adds htmlgraph session restore <sid> that extracts .tar.gz back into .htmlgraph/sessions/ so the indexer picks it up on next replay.
Why
Without SessionEnd SIGTERM, collectors either persist past session lifetime (waiting for a launcher SIGTERM that may never fire after launcher SIGKILL), or get reaped by OS cleanup with spans still in their batch buffer — losing them. Without retention, .htmlgraph/sessions/ grows unbounded over months. Using stdlib gzip (rather than zstd) keeps the retention pipeline dependency-free — compression ratio on NDJSON log files is secondary to minimizing new dependencies at this scale. Restoring an archived session is supported so lineage queries over historical transcripts still work. This slice closes the correctness gaps named in Q3 (primary + secondary drain layers) and the operational gaps noted in the design discussion.

Haiku (design review)

Slice Assessment
  • S1 SignalSink interface extraction is well-scoped and foundational. Low risk, done_when criteria are testable.
  • S2 Idle-timeout safety net from Q3 needed scoping — added to this slice in revision so all three Q3 layers are now implemented.
  • S3 Launcher changes are the critical path. 3s stdout-scan timeout reasonable. Stdout-capture pattern clarified in revision to reuse serve_child.go:70 format.
  • S5 Checkpoint partial-write recovery needs definition. Revision specifies: corrupted checkpoint falls back to offset 0, full replay; INSERT OR IGNORE makes this idempotent (verified A8).
  • S6 Rename-only change carries low risk; regression test coverage mandatory. File list expanded in revision to include claude_env.go.
  • S7 Retention archive format changed to .tar.gz (stdlib) from .tar.zst in revision — avoids new dependency.
Gaps and Missing Considerations
  • Gap Windows support — Setpgid:true is POSIX-only. Addressed via explicit platform-scope constraint (Linux/macOS only this plan).
  • Gap Indexer observability — no way for user to learn when indexer stalls. Addressed via new Q7 (q-indexer-observability).
  • Gap Spans emitted during launcher handshake window — before endpoint is set — may be dropped by Claude Code SDK. Accepted as a tolerable gap; retry behavior depends on upstream SDK.
  • Gap Schema evolution on UnifiedSignal JSON format — if v2 adds fields, indexer replay of old NDJSON should tolerate unknown fields. Implicitly handled by JSON field-ignoring semantics; not elevated to a question.
Alternative Approaches
  • Alt Single-process multi-session receiver (keep receiver in serve, multiplex by session-id tag) would avoid subprocess complexity but keep shared-contention problem. Explicitly rejected — loses lifecycle independence.
  • Alt Indexer as separate process (not inside serve) would decouple dashboard uptime from indexer. Deferred — serve-embedded is simpler for phase 1.
