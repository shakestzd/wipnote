Plan: Agentic-eng gaps: retrieval + handoff analytics + notifications

Plan for the top-2 gap cluster identified in the LangChain agentic-engineering review: (1) upgrade HtmlGraph's retrieval layer from ripgrep keyword matching to SQLite FTS5 full-text search over plans, specs, sessions, and features, giving the work graph a real query surface; and (2) add a durable work-item transition log plus handoff-dwell-time analytics plus outbound webhook notifications, so the coordination bottlenecks the article warns about become measurable and actionable.

Problem

HtmlGraph's observability spine (causal lineage, OTel receiver, session ingestion) is a genuine strength, but two gaps limit its "agentic engineering" claim. (1) Retrieval is ripgrep-based keyword matching with no tokenization beyond whitespace split (cmd/htmlgraph/relevant.go:231-247). No FTS index, no ranking beyond additive hit counts plus status bonuses, no way to find "the prior feature about X" when X is not a literal substring. Zero embedding infrastructure exists in the Go binary today. (2) Handoff analytics: state transitions on features, bugs, and plans are not recorded — only the current status and a single overwritten updated_at timestamp survive (internal/workitem/collection.go:154,173; internal/db/feature_repo.go:126). Existing `analytics velocity` buckets by updated_at and conflates completion with any later re-touch. Dwell time per transition cannot be computed from current data. (3) Notifications: no outbound integration of any kind — no webhook emitters, no event bus, no Slack/Teams/Discord adapter. Writes are point-to-point through writeNode. The article's bottleneck insight (agents accelerate coding; humans at handoff points become the new bottleneck) is invisible because there is neither measurement nor notification.

Goals

  1. **Semantic-lite retrieval** — `htmlgraph relevant <query>` returns ranked results for queries that are not literal substring matches, using SQLite FTS5 over features, messages, plans
  2. **Durable transition log** — every work-item status transition writes a row to a new `work_item_transitions` table with from/to/at/actor; backfill not required but forward-coverage is complete
  3. **Handoff dwell-time analytics** — `htmlgraph analytics handoff` and a dashboard pane report p50/p95 dwell per transition per work-item type
  4. **Outbound notifications** — configurable webhook-out on transition events, with a Slack formatter as the first concrete adapter
  5. **No regression to local-first posture** — no mandatory network dependency; FTS5 is embedded, webhooks are opt-in

Constraints

  • SQLite driver is `modernc.org/sqlite` — FTS5 support must be verified (critique will check this)
  • No existing event bus or observer pattern in the app — emission path must be added without breaking the point-to-point write path
  • Secrets handling: the project has no keyring or encrypted-file pattern today; env-var interpolation in config is the minimum viable approach
  • Backward compatibility: `htmlgraph relevant` is called by skills and cannot change its CLI contract; new behavior must be drop-in
  • Work-item writes all funnel through `internal/workitem/writeNode` — this is the single hook point for index-update and event emission
  • Transition logging must cover PlanCollection override paths (internal/workitem/plans.go:153 Start, :171 Complete) and the draft->review transition site (internal/workitem/plan_yaml_extras.go:293), not just base Collection.Start/Complete — plans bypass the base methods and will be missed if only the base is wired
  • FTS index updates should be hooked at WriteNodeHTML (internal/workitem/htmlwriter.go:74), the single terminal write function shared by all work-item types, rather than at per-type writeNode overrides — this avoids the plans.go override path missing the hook

#1 FTS5 index over work-graph content (features, messages, tracks, plans) M Med Pending
What
Add a SQLite FTS5 virtual table `content_fts` with columns (source_table, source_id, title, body) and triggers/helpers that keep it in sync with: `features` (title + description), `messages` (content; chunked if larger than 8KB), `tracks` (title + description), plus a Go-side indexer for plan YAML content (since plans are a composite of YAML + HTML + DB rows). Touches: internal/db/schema.go (new migration), internal/db/content_fts.go (new — indexer/upsert/delete helpers), internal/planyaml/io.go (post-Save FTS update), internal/workitem/htmlwriter.go — hook post-WriteNodeHTML at its single terminal write point (line 74), which covers all work-item types including the plans.go override path, internal/ingest/parser.go (insert into FTS when writing messages rows). Verify `modernc.org/sqlite` FTS5 support at migration-apply time — if FTS5 is unavailable, the migration must surface a clear error and the critique must confirm availability before this slice is accepted.
Why
Semantic-lite retrieval requires an indexed content substrate. Today the substrate is the filesystem (ripgrep over `.htmlgraph/*.html`), which precludes ranking beyond hit-counting and does not cover session transcripts at all. FTS5 is the minimum-viable index that preserves local-first operation (no network, no ONNX runtime, no external API).
Files
internal/db/schema.go, internal/db/content_fts.go, internal/planyaml/io.go, internal/workitem/htmlwriter.go, internal/ingest/parser.go, internal/workitem/plans.go
Done when
  • A startup/migration probe runs SELECT fts5_version() before any DDL; if FTS5 is unavailable the migration fails with a clear error naming the modernc.org/sqlite version constraint
  • Running the migration on a fresh DB creates `content_fts` and a smoke test confirms FTS5 tokenization works (stemming and phrase search)
  • Editing a feature's description via `htmlgraph feature edit` causes the new text to be findable via `SELECT ... FROM content_fts WHERE content_fts MATCH ?` within the same process lifetime
  • Running `htmlgraph ingest` on a JSONL transcript populates `content_fts` rows for every `messages` row written
  • A reindex helper `htmlgraph reindex --fts` rebuilds the index from canonical sources without duplicating rows
  • FTS5 unavailability is detected at migration time with an error that names the workaround

#2 Upgrade `htmlgraph relevant` to FTS5-ranked with ripgrep fallback M Med Pending
What
Replace the ripgrep-over-HTML backend in cmd/htmlgraph/relevant.go with an FTS5 `SELECT ... ORDER BY bm25(content_fts)` query. Preserve the existing status-bonus and commit-trailer-score layers: query FTS5 to get candidate IDs and base score, then apply the same weightCommitItem, in-progress, and done-penalty adjustments that exist today. Keep ripgrep as a fallback path only for file-path and SHA queries (which already skip tokenization). For natural-language queries, FTS5 is primary. CLI contract unchanged: `htmlgraph relevant <query>` prints the same output format (stable for plugin skills). Add `--explain` flag that exposes per-result score breakdown for debugging.
Why
Slice 1 gives us an index; slice 2 makes it useful. Without this, the FTS table is dark infrastructure. The kept scoring layers preserve causal signal (commits attributing an item, in-progress boost) that pure content similarity would lose.
Files
cmd/htmlgraph/relevant.go, cmd/htmlgraph/relevant_test.go
Done when
  • `htmlgraph relevant 'authentication failure'` returns ranked features/bugs even when none contain both words as a substring (stemming on)
  • A feature whose description contains `auth` ranks above one that only appears in a commit trailer, unless commit-trailer count is high enough to overcome the BM25 gap (scoring hierarchy documented)
  • `htmlgraph relevant <sha>` and `htmlgraph relevant <file-path>` still use the existing code paths (no regression)
  • `htmlgraph relevant --explain <q>` prints the score decomposition
  • Output format for non-explain mode is byte-identical to today's when the same top-N items would have been returned

#3 Work-item transition log (schema + write path) M Med Pending
What
Add a `work_item_transitions` table with columns (id INTEGER PK, work_item_id TEXT, work_item_type TEXT, from_status TEXT, to_status TEXT, at DATETIME, actor TEXT, session_id TEXT, metadata JSON). Wire inserts from every status-change site: internal/workitem/collection.go Start/Complete; internal/workitem/edit.go SetStatus+Save; cmd/htmlgraph/plan_finalize_yaml.go status = finalized; internal/planyaml/ wherever draft -> review is set; cmd/htmlgraph/workitem.go runWiSetStatus (covers features/bugs/spikes). `actor` is sourced from HTMLGRAPH_AGENT_ID / HTMLGRAPH_SESSION_ID when present; `cli-user` otherwise. Insertion happens in the same DB transaction as the status update to guarantee atomicity — if the table is absent or the insert fails, the status change still completes and the failure is logged (new behavior is non-blocking). CRITICAL: PlanCollection.Start (plans.go:153) and PlanCollection.Complete (plans.go:171) OVERRIDE Collection.Start/Complete and bypass the base. The override path must also write transitions. Similarly, the draft->review transition is set at plan_yaml_extras.go:293 via a direct call that does not go through Start/Complete — it must also emit a transition row. Alternative hook point worth considering: wire transitions inside internal/db/feature_repo.go:126 UpdateFeatureStatus so every status write is captured regardless of caller, passing actor/session_id as explicit parameters from each call site.
Why
Without a durable transition log, handoff-dwell-time is not computable. updated_at is overwritten on every mutation; completed_at exists in the DDL but is not written today (internal/db/feature_repo.go:126). Adding this is the one piece of infrastructure every downstream handoff analytic requires.
Files

Done when
  • Starting a feature creates a transitions row with from=todo, to=in_progress, at=now, actor=<env>
  • Completing a feature creates a transitions row with from=in_progress, to=done
  • Plan finalize creates a transitions row for from=<draft|review>, to=finalized
  • Migration is forward-only; existing DBs without the table have the table created without backfill (documented limitation)
  • A status-change is never rolled back due to a transitions-insert failure — failures are logged but non-fatal
  • Plan lifecycle transitions are captured via ALL paths: PlanCollection.Start/Complete overrides (plans.go:153,171), draft->review (plan_yaml_extras.go:293), draft->finalized / review->finalized (plan_finalize_yaml.go)
  • A fuzz test or coverage assertion confirms no UpdateFeatureStatus call site is missing a transition emission
Test Strategy

#4 Handoff dwell-time analytics (CLI + dashboard pane) M Low Pending
What
Add `htmlgraph analytics handoff [--type feature|bug|plan|all] [--since 30d]` that queries work_item_transitions joined with features (for type) and reports: (a) per transition edge (e.g. todo->in_progress), count + p50 + p95 dwell duration over the window; (b) per work-item type, aggregate backlog age (time since latest transition for items currently in a given state); (c) stale-work detector: items in in_progress longer than p95 for their type. Dashboard: add a "Handoffs" tab (sibling to the existing Metrics tab in cmd/htmlgraph/dashboard/js/app.js:746) that renders a timeseries of dwell p50/p95 per edge. Read data via a new /api/analytics/handoff endpoint.
Why
The LangChain article's sharpest diagnostic is that agent acceleration exposes handoff bottlenecks. Measurement is prerequisite to surface that signal. Without this slice, the transition log from slice 3 is dark data.

#5 In-process event bus with transition events as first publisher S Low Pending
What
Add a minimal in-process pub/sub at internal/events/bus.go: a `Bus` interface with Publish(event Event) and Subscribe(topic string, handler func(Event)), backed by an in-memory goroutine pool. Events have (topic, payload any, at time.Time, actor, session_id). First event type: `workitem.transition`, published from the same code sites that write the transitions row in slice 3. Transitions-row insert is synchronous and atomic; event publish is async and fire-and-forget. No built-in subscribers yet — slice 6 (webhook adapter) is the first.
Why
Notifications require an emission surface. Writes today are point-to-point (research confirmed zero event bus). Hooking notifications directly into writeNode would tangle concerns; a bus lets us add/remove adapters without touching write paths. Kept minimal (no persistent queue, no at-least-once delivery) — this is local coordination, not a message broker.
Files
internal/events/bus.go, internal/events/bus_test.go, internal/workitem/collection.go, internal/workitem/edit.go
Done when

  • Subscriber failures/panics are isolated (logged, do not affect other subscribers or the publishing code path)
  • Bus is constructed once and accessible to both cmd/ and internal/ packages without circular imports
  • Event publish latency is less than 1ms on the write-path p95 (fire-and-forget)

#6 Outbound webhook adapter + config surface (native format only; Slack formatter gated by q-slack-priority) M Med Pending
What
New `.htmlgraph/config.json` (mode 0600) with a `notifications` schema containing a list of webhooks: each entry has name, url, format (native or slack), events (list of topics), and an optional filter object (e.g. to_status: [in_progress, done]). URL values support ${ENV_VAR} interpolation at load time (secret indirection without storing tokens in the file). Config loader at internal/config/config.go with strict schema validation. Webhook adapter at internal/events/webhook.go subscribes to workitem.transition via the slice-5 bus and POSTs the configured payload. Two formatters: `native` (htmlgraph JSON schema, documented) and `slack` (text: "feat-XXXXX — Title: <from> -> <to> (actor)"). The Slack formatter is included in this slice only if q-slack-priority is answered 'include-slack-now'. Default answer defers Slack to a follow-up plan, shipping just the native formatter in this plan — reduces scope for a solo-dev-first release and validates the event surface before building adapter variants. Retry once on 5xx with exponential backoff, drop on 4xx. Startup registers webhook subscribers based on config; missing config means no subscribers (no-op). `htmlgraph notify test <name>` sends a synthetic test event to the named webhook for smoke-testing.
Why
Completes the notification loop: transitions -> bus -> webhook -> Slack. Without a formatter, the event bus is dark. Slack is the most common team chat target; the `native` format gives users a stable schema to build custom adapters against. Local-first is preserved: no webhooks configured equals no network traffic.
Files
internal/config/config.go, internal/events/webhook.go, cmd/htmlgraph/notify.go, docs/notifications.md
Done when
  • `htmlgraph notify test slack-dev` posts a synthetic event and prints the response status/body
  • A real feature start triggers a Slack message matching the formatter spec
  • ${ENV_VAR} interpolation expands at load time; missing env vars produce a startup error naming the unresolved var
  • 5xx response triggers one retry; 4xx drops without retry; both log with webhook name and status
  • Absent config.json is not an error — startup proceeds with zero webhook subscribers
  • File permissions on config.json are checked (warn on >0600)
Test Strategy
Unit: config_test.go for schema + env interpolation + permission check; webhook_test.go with httptest.Server for success/5xx/4xx/timeout Integration: end-to-end test — start a feature, assert httptest.Server received a POST with expected body Regression: running htmlgraph without config.json behaves identically to today

Synthesis

Plan is directionally sound. HIGH-severity feasibility defects were found and addressed before review: (1) slice 3 was missing PlanCollection override paths, now explicitly covered; (2) FTS hook point was wrong (writeNode vs. WriteNodeHTML), now corrected. FTS5 availability is the highest remaining risk, mitigated by a fail-fast probe in slice 1 done_when. Seven open design questions for the human reviewer. Three are critical (the rest are defaults likely to be accepted): 1. q-retrieval-backend — FTS5 only (recommended) vs. local embeddings vs. external API 2. q-slack-priority — defer Slack (recommended) vs. include now vs. drop all notifications 3. q-slice-1-2-framing — keep slices split (recommended) vs. collapse into one If reviewer answers q-slack-priority = 'drop-all-notifications', slices 5+6 drop and the plan becomes retrieval (1-2) + observability (3-4) only — a cleaner scope given the solo-dev context.
