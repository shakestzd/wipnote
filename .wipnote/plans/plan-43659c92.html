Plan: Close session HTML canonical gaps

Close the three remaining gaps in the existing session HTML canonical implementation. Sessions already use HTML as canonical storage per feat-029affaa (createSessionHTML at SessionStart, appendEventToSessionHTML at PostToolUse via flock + read-modify-rewrite, finalizeSessionHTML at SessionEnd, reindex round-trip via goquery). This plan does NOT introduce the pattern — it fills the three gaps the pattern left open: (1) ingest path writes SQLite but never creates session HTML files, (2) tool calls where PostToolUse never fires stay as orphaned SQLite rows forever with no HTML entry, (3) legacy sessions created before feat-029affaa shipped have no HTML files and can't round-trip through reindex. Performance optimization (swapping flock for O_APPEND) and mid-session header refresh are deliberately out of scope — filed for separate benchmark-driven evaluation.

Problem

The existing session HTML canonical implementation (feat-029affaa, shipped on trk-97f85b3b) has three gaps that became visible during multi-project hardening smoke-testing on 2026-04-10. **Gap 1 — Ingest path skips HTML.** `cmd/htmlgraph/ingest.go:storeParseResult` writes ingested Claude Code transcripts to the `messages`, `tool_calls`, and `agent_events` tables, but never calls `createSessionHTML`. When `DiscoverSessions` scans `~/.claude/projects/<slug>/*.jsonl` and processes historical transcripts, the resulting session rows exist only in SQLite. This is the root cause surfaced by bug-a52d5bf9: planbook had 15 session rows in its database but only 3 HTML files, and the 12 orphaned rows came from ingested transcripts that bypassed the canonical HTML path. The display-side fix added a `WHERE project_dir = ?` clause to `sessionsHandler` and the ingest-side fix corrected an inverted prefix filter, but neither addressed the architectural gap: ingested sessions are still SQLite-only, which means `htmlgraph reindex` (which reads HTML files to rebuild SQLite) cannot recover them if the DB is lost. **Gap 2 — No orphan sweep for crashed tool calls.** `internal/hooks/pretooluse.go` writes an `agent_events` row with `status='started'`; `PostToolUse` later looks it up via `FindStartedEvent` to get the event_id and calls `appendEventToSessionHTML` to write the `<li>` entry. If PostToolUse never fires — tool crash, Claude Code kill, system shutdown mid-tool-call — the row stays in `status='started'` forever and the HTML file never gets the `<li>`. The session history has permanent silent holes. No sweep, no recovery, no synthetic aborted entry. **Gap 3 — Legacy SQLite-only sessions.** htmlgraph has 485 session rows in its SQLite index but only 325 HTML files in `.htmlgraph/sessions/`. The 160-row gap is sessions that were created before feat-029affaa shipped (via hook paths that only wrote SQLite) or via the ingest path (Gap 1). These sessions cannot be reindexed via `reindexSessions` because there is no HTML to parse. They show up correctly in the dashboard (SQLite-backed views) but break the invariant that HTML is canonical and SQLite is derivable. A one-time backfill would render HTML files from their SQLite data so the invariant holds going forward. All three gaps share a root cause: **the session HTML canonical pattern was shipped incrementally and the ingest + recovery + migration corners were never closed.** This plan closes them without introducing new architecture.

Goals

  1. **Ingest parity with live hooks** — `htmlgraph ingest` and the auto-ingest loop in `autoIngestOnce` render a complete session HTML file for every ingested transcript, using the same `createSessionHTML` + event-append path as live SessionStart/PostToolUse hooks. Ingested sessions become round-trippable via `htmlgraph reindex`.
  2. **Crash recovery for orphaned tool calls** — tool calls where `agent_events.status='started'` has been set but no PostToolUse ever completed get a synthetic `<li data-status="aborted" data-reason="no-post-hook">` entry written to the session HTML file after a configurable timeout. The SQLite row transitions to `status='aborted'`. Sessions that end mid-tool-call recover gracefully — no missing events, no permanent holes.
  3. **Legacy session backfill** — one-time migration that finds every session row with no corresponding HTML file, renders it via the same template path from Goal 1, and writes the resulting HTML to `.htmlgraph/sessions/<id>.html`. After the migration, every session row has a canonical HTML counterpart and the reindex round-trip works for every session.
  4. **No regressions in the happy path** — existing SessionStart/PostToolUse/SessionEnd hook behavior is unchanged. The flock + read-modify-rewrite pattern for live appends stays. Existing session HTML files continue to parse correctly via the reindex path.
  5. **Acceptance via reindex round-trip** — an integration test that (a) creates a session via the ingest path, (b) nukes the SQLite DB, (c) runs `htmlgraph reindex`, and (d) asserts the rebuilt session row matches what ingest originally wrote. This proves the HTML is genuinely canonical for ingested sessions, not just a mirror.

Constraints

  • **Do NOT change the existing live-hook write path.** `createSessionHTML`, `appendEventToSessionHTML`, and `finalizeSessionHTML` in `internal/hooks/session_html.go` stay as-is. The flock-based read-modify-rewrite pattern is not being replaced. This plan extends, never rewrites, the existing canonical implementation.
  • **Performance optimization is OUT OF SCOPE.** The O(N) full-file rewrite per tool call under the flock pattern is a known tradeoff but not the problem being solved here. Swapping to `O_APPEND` with structurally-open documents is a separate piece of work that should be gated on benchmarks showing a measurable impact on real sessions. File it independently if you want it.
  • **Mid-session header refresh is OUT OF SCOPE.** The stale `data-event-count` and status badge during a running session are cosmetic for an infrequently-read artifact (HTML files are not what the dashboard queries — SQLite is). `finalizeSessionHTML` at SessionEnd updates the header correctly; mid-session is never visibly wrong in any consumer that matters today. Ignore unless a specific consumer demands it.
  • **Reuse `createSessionHTML` and `appendEventToSessionHTML` verbatim for ingest rendering.** The template, escaping discipline (`html.EscapeString`), attribute schema (`data-ts`, `data-tool`, `data-success`, `data-event-id`, optional `data-feature`), and `<ol reversed>` convention must match exactly so reindex_sessions.go's goquery parser keeps working without modification.
  • **Orphan sweep runs lazily, not via a daemon.** No long-running background process. The sweep is triggered by (a) the next hook event for the session or (b) SessionStart of a new session in the same project or (c) on demand via a CLI subcommand. This matches the existing per-event subprocess model where `env vars don't persist between hook processes` and there is nowhere for a daemon to live.
  • **Orphan sweep must be idempotent.** Running it multiple times on the same orphaned event must produce exactly one synthetic `<li>` entry, not duplicates. Use the event_id as the dedup key.
  • **Backfill migration is one-shot.** Not called on every invocation; only runs when `htmlgraph migrate sessions` (or similar explicit subcommand) is invoked. Reports count of rendered files and any errors.
  • **goquery round-trip must continue to work.** `cmd/htmlgraph/reindex_sessions.go` parses session HTML files via goquery and extracts session metadata + event list. Any HTML rendered by the ingest path or the backfill migration must parse identically to a live-hook-written file.
  • **No new dependencies.** All work uses existing packages: `internal/hooks/session_html.go` for rendering, `cmd/htmlgraph/ingest.go` for ingest plumbing, `internal/ingest` for transcript parsing, `goquery` (already a dep via reindex_sessions.go) for parsing existing files if needed.
  • **Cross-package reuse requires exports.** `createSessionHTML`, `appendEventToSessionHTML`, `finalizeSessionHTML`, the `sessionEvent` struct, and `summariseInput` are all currently unexported in `internal/hooks/` (session_html.go:31, session_html.go:112, session_html.go:193, session_html.go:19, pretooluse.go:368). `cmd/htmlgraph/` cannot call them directly without first exporting them — this is a hard compile-time barrier that slice 1 must address before any reuse is possible. The exports are a one-time renaming refactor (4 symbols + any test file updates), not a rewrite.
  • **Live and ingest HTML files are structurally identical.** Both `createSessionHTML` (live) and the new `renderIngestedSessionHTML` (ingest) produce complete self-contained HTML files with opening AND closing tags. The difference is NOT the structure but the write mechanism: live appends events one at a time via `appendEventToSessionHTML`'s flock + read-modify-rewrite (scanning for the last `</ol>` marker and inserting before it), while ingest writes all events at once. Both file shapes are parsed identically by `reindexSessions` via goquery.
  • **agent_events schema migration required.** Slice 2's sweep needs a `reason TEXT` column on `agent_events` to distinguish swept-aborted events from completed ones. The column does NOT exist today (verified against `internal/db/schema.go:96-130`). It must be added via an idempotent `ALTER TABLE agent_events ADD COLUMN reason TEXT` in `schema.go:Open()`, matching the existing migration pattern at `schema.go:41`. This is one line but is non-optional.

#2 Orphan sweep for crashed tool calls M Med Pending
What
Add a sweep function that detects `agent_events` rows with `status='started'` where no PostToolUse ever fired, emits a synthetic aborted `<li>` entry to the session HTML file, and transitions the SQLite row to `status='aborted'` with a `reason='swept'` marker. The sweep is lazy — triggered by existing hook events, not by a daemon. **(0) Schema migration — COMPLETED before plan finalization (see commit log).** Add `db.Exec("ALTER TABLE agent_events ADD COLUMN reason TEXT")` to the idempotent migration block in `internal/db/schema.go:Open()`, following the existing pattern at line 41 (`ALTER TABLE sessions ADD COLUMN title TEXT`). Landed in a pre-finalization commit as a one-line addition to the existing idempotent migration block — the sweep logic writes to this column and will error at runtime if the column is missing. The migration is idempotent (SQLite's `ALTER TABLE ADD COLUMN` fails harmlessly if the column already exists, and the existing migration pattern wraps the Exec in a swallowed error). **(1) Detection query.** New function `FindOrphanedEvents(db *sql.DB, olderThan time.Duration) ([]OrphanEvent, error)` in `internal/db/event_repo.go` (alongside `FindStartedEvent` at line 261). Returns rows where `status='started'` AND `created_at < now - olderThan`. Default threshold: 5 minutes, stored as a constant `internal/hooks/orphanThreshold = 5 * time.Minute`. The query uses the existing `idx_agent_events_session_ts_desc` index — no new index required. **(2) Synthetic append.** New function `AppendOrphanedEvent(htmlgraphDir, sessionID string, orphan OrphanEvent) error` in `internal/hooks/session_html.go`. Constructs a `SessionEvent` with tool name from the orphan, timestamp from its `created_at`, `data-status="aborted"`, and `data-reason="no-post-hook"`, then calls `AppendEventToSessionHTML` via the existing flock path. No new write mechanism — just a specialized event wrapper. On success, updates the agent_events row to `status='aborted', reason='swept'`. **(3) Idempotency via goquery dedup check.** Before appending, the sweep reads the session HTML file and checks whether an `<li data-event-id="<id>">` already exists. If yes, skip the append and just update the SQLite status to keep the row consistent. Reading the file via goquery is acceptable — the sweep is a rare path and goquery is already a dependency via `reindex_sessions.go`. **(4) Lazy trigger — all three paths (from Q1, recommended default).** Invoke `SweepOrphanedEventsForSession(sessionID)` from: (a) the end of every PostToolUse after the normal append, scoped to the current session; (b) the end of SessionStart, scoped to all sessions in the current project; (c) a new CLI subcommand `htmlgraph sweep orphaned-events [--session <id>] [--project-dir <dir>]` for explicit user-triggered sweeps. Default scope for the CLI: all sessions in the current project. Reports count of swept events. **(5) Hard cutoff at 24 hours.** Orphans older than 24h are still processed (synthetic entry + SQLite update) but logged with a warning so users know about the gap. Prevents silently sweeping ancient orphans as if they were recent crashes. **(6) SessionEnd race guard.** `FinalizeSessionHTML` (previously `finalizeSessionHTML`) does NOT use flock — it's a simple read-modify-write of header attributes. If a sweep runs during SessionEnd, the sweep's flocked append could race with the finalize's non-flocked header update. Mitigation: wrap `FinalizeSessionHTML` in the same flock acquisition as `AppendEventToSessionHTML` (open the file with `O_RDWR`, `syscall.Flock LOCK_EX`, read, modify, write, release). This is a one-line addition to an existing function and it preserves the existing semantics. This closes Gap 2.
Why
Sessions that crash mid-tool-call leave silent holes in both SQLite and HTML. Without a sweep, those holes are permanent — the session history becomes gradually less accurate over time. This slice guarantees eventual consistency: every `status='started'` row either gets a PostToolUse completion or a synthetic aborted entry within a bounded time window.

What

The migration runs once and is not invoked automatically on every CLI call. **(a) New CLI subcommand.** `htmlgraph migrate sessions [--project-dir <dir>] [--dry-run]`. #4 Reindex round-trip acceptance test M Low Pending
What
End-to-end integration test that proves HTML is truly canonical for all session types after slices 1-3 land. The test verifies: **(a) Ingest produces reindex-compatible HTML.** Ingest a fixture JSONL transcript via `htmlgraph ingest`, verify SQLite row exists, verify HTML file exists, nuke the SQLite DB, run `htmlgraph reindex`, verify the session row is rebuilt with identical agent_events count and metadata. **(b) Live hook HTML round-trips.** Simulate a SessionStart → PreToolUse → PostToolUse → SessionEnd sequence via the hook CLI commands, verify the resulting HTML file, nuke SQLite, reindex, verify the rebuilt row matches. **(c) Orphaned events are recovered.** Simulate a PreToolUse without a matching PostToolUse (direct insert into agent_events with `status='started'`), advance time past the threshold, run `htmlgraph sweep orphaned-events`, verify the HTML file has a new synthetic `<li data-status="aborted">` entry, nuke SQLite, reindex, verify the rebuilt row shows `status='aborted'`. **(d) Migration handles orphans.** Create a tmpdir project with SQLite rows and NO HTML files (simulating legacy state), run `htmlgraph migrate sessions`, verify HTML files are created, nuke SQLite, reindex, verify all sessions rebuild. **(e) Concurrent writers.** Spawn N goroutines simulating concurrent hook subprocesses calling `appendEventToSessionHTML` on the same session file. Assert the resulting file is (1) well-formed and parseable by goquery AND (2) contains ALL N events — not just 'no crash' or a subset. This is the load-bearing acceptance criterion for R1 (concurrent flock writers landing all events). Spawn 20+ goroutines across three writer types — ingest render, live PostToolUse append, and sweep append — all targeting the same session HTML file. Verify every single event's `data-event-id` is present in the final file. This verifies the existing flock path still works after any refactoring in slices 1-3. All tests under `//go:build integration` build tag so they don't slow the default `go test ./...` run. CI runs them in a separate job.
Why
Without this test, slices 1-3 could ship while subtly breaking the round-trip invariant. The entire point of the HTML-canonical story is `reindex` can rebuild SQLite from HTML. This test is the acceptance criterion that proves it holds for every session type: ingested, live-hook, aborted, backfilled.
