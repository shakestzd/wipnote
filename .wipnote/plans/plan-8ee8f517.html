Plan: Mount NDJSON to SQLite indexer inside per-session otel-collect; close contention holes

Make the documented "NDJSON canonical, SQLite derived" contract operational by mounting the existing internal/otel/indexer as a goroutine inside the per-session otel-collect subprocess. Address four contention/throughput issues the researcher surfaced: (1) hooks bypass the indexer with direct INSERTs into otel_signals, (2) indexer WriteBatch is O(N) per signal, (3) gracefulShutdown has no indexer drain inside its 3s window, (4) internal/registry/registry.go bypasses db.Open and inherits no busy_timeout. End state: dashboard read paths see live span data again, single writer per session, NDJSON remains source of truth, SQLite is a rebuildable derived index.

Problem

The per-session otel-collect subprocess writes spans only to NDJSON files at .htmlgraph/sessions/<id>/events.ndjson. The indexer that bridges NDJSON to SQLite (internal/otel/indexer) has zero production callers — it was pulled out of _serve-child to fix SQLITE_BUSY contention (bug-4697c62c) and never re-mounted. Consequence: every read path against otel_signals (dashboard /api/otel/* endpoints, htmlgraph session list, the internal/otel/materialize layer) sees stale or empty data because nothing writes live spans into SQLite. The documented contract from feat-b6479241 is technically intact but operationally broken. Compounding: hooks at internal/hooks/transcript_user_prompt.go:191 and transcript_assistant.go:298 write directly to otel_signals via raw SQL, bypassing the indexer; indexer.go:228-231 calls WriteBatch once per signal; otel_collect.go:168-172 gracefulShutdown has a hard 3s deadline with no indexer drain; internal/registry/registry.go:216 bypasses the canonical db.Open helper and never inherits busy_timeout=5000.

Goals

  1. **Live span visibility** — dashboard and CLI reads against otel_signals return current data within one indexer tick (~500ms) of OTLP receive
  2. **Single writer per session** — exactly one process holds a SQLite write handle for a given session's OTel data
  3. **Documented contract operational** — NDJSON files are source of truth; SQLite is rebuildable via the indexer or htmlgraph reindex
  4. **Throughput floor** — indexer sustains 1000 signals/sec on a 4-core dev container without becoming the bottleneck
  5. **Shutdown safety** — SIGTERM drains the indexer cleanly within an extended budget; orphaned signals recover via checkpoint replay on next collector launch

Constraints

  • Must not change OTLP wire format or the WriterCloser / SignalSink interfaces beyond widening WriteBatch to accept a batch
  • Must preserve idempotent replay — checkpoint reset → full replay → INSERT OR IGNORE keeps DB consistent
  • Must coexist with work-item HTML hooks that also write to the canonical DB
  • Per-session collector remains the only OTLP receiver — no central daemon
  • Cannot break in-flight OTel sessions on rollout — handshake protocol stays the same
  • Build break in internal/otel/indexer/indexer_test.go (stale internal/otel/receiver import) must be fixed before any indexer-touching slice can be tested — addressed by slice-8
  • errgroup is not allowed: golang.org/x/sync is not in go.mod requires; use sync.WaitGroup + error channel

#1 Mount indexer goroutine inside otel-collect M Med Pending
What
In cmd/htmlgraph/otel_collect.go runOtelCollect, spawn internal/otel/indexer.New(...).Start(ctx) as a goroutine after the HTTP server starts and before awaitShutdown blocks. Coordinate goroutine lifecycle via a `sync.WaitGroup` plus an error channel (stdlib only — `golang.org/x/sync/errgroup` is not in `go.mod` requires) so server + indexer share lifecycle and a cancellation propagates to both. The indexer's *sql.DB injection is decided in q-db-handle-injection (recommended: collector opens via db.Open and injects via WithDB). Pass the indexer the `.htmlgraph/` subdirectory path: `htmlgraphDir := filepath.Join(projectDir, ".htmlgraph")` — `indexer.New` does NOT accept the project root. DB handle injection follows q-db-handle-injection (recommended option-b): collector calls `db.Open(<canonical-db-path>)` and passes the handle to `indexer.New(...).WithDB(db)`. The collector also adds `internal/db` to its imports for the first time.
Why
The indexer code in internal/otel/indexer exists and is unmounted; this is the missing wiring that makes spans visible in SQLite. Without it, every /api/otel/* endpoint and the dashboard return empty.

#2 Extend gracefulShutdown to drain the indexer S Low Pending
What
In gracefulShutdown at cmd/htmlgraph/otel_collect.go:168, wait on the indexer goroutine via the WaitGroup before returning. Lift the hard 3s deadline to a configurable budget — default 10s, overridable via HTMLGRAPH_OTEL_DRAIN_SECS — since drain may flush a backlog of NDJSON. Order of teardown: HTTP server Shutdown → cancel indexer ctx → wait WaitGroup → snk.Close().
Why
Without this, SIGTERM during active OTLP traffic orphans signals that have landed in NDJSON but not yet been indexed. Checkpoint replay covers the gap only if the same session is re-launched; for short-lived test sessions or ephemeral CI runs, that does not happen.
Files
cmd/htmlgraph/otel_collect.go, cmd/htmlgraph/otel_collect_test.go
Done when
  • SIGTERM during a 100-signal burst results in zero unindexed signals after the drain window
  • HTMLGRAPH_OTEL_DRAIN_SECS env var overrides the default budget
  • If drain exceeds budget, the collector logs the count of signals left in NDJSON and exits non-zero
  • Load test TestOtelCollect_GracefulShutdown_Under10kSignals confirms 10s drain is sufficient on a 4-core dev container
Test Strategy
Integration: TestOtelCollect_GracefulShutdown_DrainsIndexer — emit 100 OTLP spans, send SIGTERM immediately, await process exit, query otel_signals, assert all 100 present. Unit: TestGracefulShutdown_RespectsDrainBudget — set HTMLGRAPH_OTEL_DRAIN_SECS=1 and verify the timeout fires.

#3 Migrate hook direct-writes to NDJSON M High Pending
What
Remove the direct INSERT OR IGNORE into otel_signals from internal/hooks/transcript_user_prompt.go:191 and internal/hooks/transcript_assistant.go:298. Replace with NDJSON append to the session's events.ndjson — Verified by the feasibility critic that hooks have `sessionID` and `projectDir` in scope at the INSERT call sites (transcript_user_prompt.go:191 and transcript_assistant.go:298 receive both via the hook event payload and ResolveProjectDir helper). Implementation per q-write-coordination (recommended option-a): hooks call `ndjson.New(projectDir, sessionID).WriteBatch(...)`. The per-session collector's indexer picks them up on next tick. Hidden work flagged by the feasibility critic: the current INSERT writes raw column values (signal_id, span_id, ts_micros, etc.); the NDJSON line format is a full `otel.UnifiedSignal` struct. The mapping from raw transcript record fields to UnifiedSignal must be worked out explicitly during implementation — this is not a one-line replacement.
Why
Eliminates the dual-writer contention that started this whole story (bug-4697c62c). Hooks become NDJSON producers, indexer becomes the sole SQLite writer. Without this slice, the indexer goroutine and the hook processes will fight for the canonical DB write lock under load.

  • A transcript hook invocation in a fresh session produces a row in otel_signals within one indexer tick after the hook returns
  • Hook latency under load (1k invocations) regresses by less than 5% versus the direct-SQLite baseline
Test Strategy

#4 Batch indexer WriteBatch calls S Med Pending
What
Change internal/otel/indexer/indexer.go:223-235's `writeParsedBatch` — accumulate signals into a slice and call WriteBatch once per tick per session. Apply the granularity decided in q-batch-granularity (recommended: cap at N=500 signals per txn, configurable via env). Verify before changing the SQLite sink: read `internal/otel/sink/sqlite/sqlite.go` WriteBatch and confirm whether it executes the slice in a single transaction or loops per signal. If it loops, the sink fix is in scope; if it already batches, only the indexer changes.
Why
A 200-signal NDJSON tick currently produces 200 sequential SQLite INSERTs. With batching it is one transaction. Without this, the indexer becomes the new throughput bottleneck once it is mounted.

#5 Apply WAL pragma to registry.OpenReadOnly path S Low Pending
What
internal/registry/registry.go:215-220 (`OpenReadOnly`) already sets `_busy_timeout=5000` in its DSN string — busy_timeout is NOT the gap (feasibility critic A4 falsified the original plan's framing). The actual gap is that this path skips the WAL mode pragma and other journal-mode logic in `internal/db/pragmas.go`. Refactor `OpenReadOnly` to either route through `db.Open(...)` (which applies all pragmas including WAL selection) or to explicitly invoke `BuildPragmas` from `internal/db/pragmas.go` after opening.
Why
Silent contention hole — with the indexer goroutine now holding writes, any registry read during high write traffic can hit SQLITE_BUSY instantly because this caller did not inherit the standard busy_timeout DSN pragma.

Synthesis

Plan is architecturally sound but had four factual errors and missed a hard build-break precondition. Revisions: (1) added slice-8 to fix the indexer test build break, (2) corrected slice-4 scope (no interface widening), (3) corrected slice-5 framing (WAL gap not busy_timeout gap), (4) removed non-existent file from slice-3 and added UnifiedSignal mapping note, (5) switched slices 1+2 from errgroup to sync.WaitGroup, (6) added load-test requirement to slice-2 and polling to slice-7. After these fixes the plan is ready for human review.
