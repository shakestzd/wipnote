Description

Symptom: .htmlgraph/logs/serve-46c1de7d.log fills with 'database is locked (5) (SQLITE_BUSY)' warnings — 156+ entries observed today — for both auto-ingest message upserts and hook tool_call inserts. Dashboard API queries intermittently fail reads because the DB is exclusively locked. Result: silent data loss in the canonical read index.\n\nRoot cause: journal_mode=DELETE (forced by the fstype-unsafe detection on virtiofs — correct choice for avoiding the WAL-SHM mmap SIGBUS) serializes all writers through a single exclusive lock. With three concurrent writer paths (hook binary per tool_call, OTel receiver batching spans, auto-ingest transcriber replaying JSONL), busy_timeout=5000 isn't enough — many begin-deferred transactions fail to upgrade to exclusive immediately.\n\nCompounding factor observed: launcher's ensureServeForOtel auto-spawns a second supervisor when the user also starts one manually. Double the writers, double the contention. Worth a separate check-and-skip fix in ensureServeForOtel (probe for any supervisor on any port, not just the expected one), but the underlying contention remains even with a single supervisor.\n\nCandidate fixes (pick one or combine):\n\n1. BEGIN IMMEDIATE for all writes — busy_timeout actually blocks on exclusive-upgrade rather than failing immediately on the deferred path. Small diff, moderate improvement, still serializes writers.\n\n2. Single-process write queue in Go — all DB writes funnel through one goroutine behind a channel. The OTel writer already uses SetMaxOpenConns(1); extend the same discipline to hooks and auto-ingest. Medium refactor.\n\n3. Relocate the read-index DB to a Docker volume on ext4 (canonical HTML stays in the virtiofs bind mount). WAL-mode then works without the SIGBUS risk, writers don't block readers, multi-writer concurrency returns. Largest change but architecturally correct. devcontainer.json adds a named volume like htmlgraph-db:/workspaces/htmlgraph/.htmlgraph/.db/ and DBPath resolution gains a small override. One rebuild to land.\n\nRecommendation: ship (1) as an immediate mitigation; plan (3) as the durable fix. Files impacted: internal/db/schema.go (transaction setup), internal/otel/receiver/writer.go, cmd/htmlgraph/claude_serve_autostart.go (duplicate-supervisor detection), .devcontainer/devcontainer.json (for option 3).
