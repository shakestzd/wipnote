WriteBatch uses DEFERRED transaction instead of IMMEDIATE — masks lock contention

Description

Latent correctness bug found while diagnosing bug-ac25cf0a. internal/otel/receiver/writer.go:208 opens a transaction with sql.TxOptions{} (zero value = DEFERRED) but the comment at line 218-220 claims to 'Upgrade the tx to an IMMEDIATE writer lock'. The follow-up SELECT 1 at line 221 only acquires SHARED, not RESERVED — so the tx remains DEFERRED through any subsequent INSERT, with the SHARED→RESERVED upgrade racing other writers. This is the underlying contention source that triggered bug-ac25cf0a's PRAGMA lock collision; bug-ac25cf0a's fix (skip journal_mode SET when already-WAL) eliminated the visible symptom but the writer.go race window remains. Fix: either pass an actual IMMEDIATE BEGIN via raw conn.ExecContext, or document that the modernc.org/sqlite driver does not honor TxOptions for upgrade semantics and use BEGIN IMMEDIATE explicitly. Two writer goroutines in _serve-child (otelreceiver.NewWriter for OTLP + idxWriter for indexer, both MaxOpenConns=1) double the collision window — separate but related issue worth keeping in mind during the fix.
