Description

Latent gap discovered during bug-d2d3fb3f verification (2026-04-09). The agent_events table schema declares a parent_agent_id TEXT column (internal/db/schema.go) intended to capture agent-to-agent spawning relationships, but nothing in the hook pipeline writes to it. Empirical evidence: queried all 6 events from a parallel probe dispatch — every row has parent_agent_id NULL, even the tool_calls inside subagents whose spawning task_delegation event was clearly identifiable. Impact: for multi-level subagent nesting (top-level → A → B), the graph cannot materialize the A→B edge. You can reconstruct it by walking parent_event_id chains back through task_delegation events, but it's not materialized as a direct agent-to-agent edge. Lineage tracking via lineage_traces and sessions.parent_session_id works at session-to-session level but cannot represent intra-session agent nesting because all subagents share a session_id. Fix: in internal/hooks/subagent_start.go and the tool-use event path, when recording an event fired inside a subagent, look up the spawning agent_id by walking parent_event_id back to the nearest task_delegation row and populating parent_agent_id with its agent_id. Alternative: populate parent_agent_id at write time if HTMLGRAPH_PARENT_AGENT_ID env var is available from the subagent's env. Not a blocker for bug-d2d3fb3f; filed as separate follow-up. Low priority for now — the parent_event_id chain is sufficient for most queries.
