Description

htmlgraph CLI commands intermittently fail at startup with: open project: open database: applying pragmas: applying PRAGMA journal_mode: database is locked (5) (SQLITE_BUSY) Repro: in a working shell, run any htmlgraph CLI that opens the SQLite project DB shortly after another long-running htmlgraph process. Observed today on: htmlgraph context-pack htmlgraph plan show (truncated output to header-only — confirms partial open / failure path that does not surface a clean error to the user) htmlgraph bug create ... (transient; succeeds on retry) Symptoms (TWO failure modes — second one is worse): 1. Loud failure: full stderr 'database is locked' and non-zero exit. Acceptable; user retries. 2. Silent partial-open: command appears to succeed but emits only the cached header section and silently drops the body (observed on plan show). The user only notices because the output looks truncated. Partial-open should fail loudly, not return half a result. Likely cause: the SQLite open path applies PRAGMA journal_mode without retry/backoff under contention. Combined with the existing concern in plan-3bbc313f slice-1 about WAL falling back to DELETE on Codespaces fs, brief lock contention turns into hard failures or silent truncation. Suggested fix scope: internal/storage/dbpath.go and internal/db/pragmas.go — wrap PRAGMA application in a busy_timeout (PRAGMA busy_timeout=N before journal_mode) and/or a bounded retry loop; ensure partial-open paths return an error rather than degrading to a header-only result; audit plan show specifically for the silent-truncation case. Related: - bug-ab377859 (otel-collect handshake timeout): same root cause class, different victim. - plan-3bbc313f slice-1, slice-2 (auto-recover empty/corrupt DB) — this bug is a sibling failure mode that slice-2's quarantine path would need to handle. Worth folding into the slice-1 RCA. Severity: medium — does not block work but undermines confidence in CLI output until partial-open is fixed.
