Evidence from DB (trace d3fa3d2fd8600a97): subagent-emitted Edit spans 6793b5c483f3666c and fbbe292a126f251f have parent_span=interaction, not parent_span=Agent, even though they were produced inside the subagent's execution window. Subsequent spans in the same trace (Bash, tool_execution, etc.) correctly parent to the Agent span (433e504d97f50699). Pattern: spans emitted early in the subagent lifecycle (before Claude Code fully establishes Agent-span OTel context) leak upward to the interaction scope; spans emitted later correctly inherit the Agent context. Dashboard cannot fix this in _indexSpans because the parent_span pointer itself is wrong. Workaround options: (1) receiver-side re-attribution by overlap window — if a tool span's [start, end] is contained within some Agent span's [start, end] in the same trace, re-parent under the Agent; (2) re-attribution by htmlgraph.agent_id resource attribute — for new dispatches, every subagent-emitted span carries agent_id as a resource attr; any span whose resource agent_id matches an Agent span AND whose parent_span points elsewhere is provably mis-parented, re-parent in internal/otel/receiver/writer.go. Option 2 is cleanest but only works post-feat-e1efb972 (once OTEL_RESOURCE_ATTRIBUTES injection is in effect). Likely upstream: Claude Code internal trace context propagation around subagent invocation.
