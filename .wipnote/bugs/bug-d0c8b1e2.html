Description

Symptom: PreToolUse:Bash blocks read-only filesystem inspection commands with 'Research is required before writing code via Bash. Read existing code first: use Read, Grep, or Glob tools.' even when the Bash call is purely inspecting agent-team state (e.g. 'ls -la ~/.claude/tasks//' or 'cat' of agent output files). Source of message: internal/hooks/yolo_guard.go (confirmed by repo grep). Root cause class: the guard heuristic over-classifies any Bash invocation as 'writing code' regardless of whether it actually writes. Pure inspection commands (ls, cat, stat, lsof, find with no -exec, grep on filesystem outside the project tree) get blocked. Wrong-footed fallback advice: the message tells the user to use Read/Grep/Glob instead. Those tools cannot read paths outside the project root (such as ~/.claude/tasks/), so the suggested alternative is unusable for the exact case where the guard fires. Distinct from bug-62f14f8c (storage-layer dual-DB-paths). This bug is purely a heuristic-classification issue in the guard, not a data-staleness issue. Investigation needed: - Read internal/hooks/yolo_guard.go and understand the current 'is this writing code?' classifier. - Identify the over-trigger: does it allowlist specific commands, denylist specific commands, or just block everything not on a research-tool list? - Decide the right model: explicit allowlist of read-only inspection commands (ls, cat, stat, file, lsof, fuser, find without -exec, head, tail, wc, grep on absolute paths outside project), or content-based detection of write intent. - Update the fallback message: when the target path is outside the project root, do not suggest Read/Grep/Glob — those tools do not reach there.
