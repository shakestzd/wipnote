OTel silently drops spans on SQLITE_BUSY + parallel launcher spawns collide on :8080

Done High Priority

Implemented In:

Part Of:

Description

Two coupled root causes for the user-visible OTel outages: (1) writer returns early without retry when SQLite reports database-is-locked; serve-64968b38.log shows continuous SQLITE_BUSY spam during contention windows. (2) New Claude sessions in parallel terminals fire SessionStart which calls ensureServeForOtel → spawnDetachedServe, which races against an already-running serve and logs 'listen tcp 127.0.0.1:8080: bind: address already in use' (see serve-auto.log). The failed spawn adds load and may momentarily write to the DB before exiting. Fix: (a) add bounded retry with exponential backoff in the writer for SQLITE_BUSY; (b) harden ensureServeForOtel's port probe to use a lockfile or PID file under .htmlgraph/ to detect an already-running parent serve reliably, not just a TCP probe; (c) optional: writer should batch multi-span INSERTs in a single transaction so contention windows shrink.
