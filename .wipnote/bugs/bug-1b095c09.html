Description

Dashboard shows every subagent delegation row labeled GENERAL-PURPOSE / agent_id=codex, regardless of the named agent invoked via Task/Agent (htmlgraph:researcher, htmlgraph:haiku-coder, htmlgraph:sonnet-coder). Trace at /tmp/htmlgraph-hook-trace.jsonl confirms Claude Code sends the correct values in raw_payload (agent_id='', agent_type='htmlgraph:') but the 'resolved' object the hook handler sees has agent_id='codex' and effectively empty agent_type (falling through to the 'general-purpose' default at internal/hooks/subagent_start.go:29). Some resolver step between stdin JSON decode and handler invocation mutates agent_id. Likely suspects: (1) HTMLGRAPH_AGENT_ID or similar env leaked from an earlier 'codex exec' process in the parent shell chain — there IS a live codex process in the container from prior delegation; (2) an agent.ResolveAgentID() or similar helper that prefers env/harness context over raw_payload; (3) resolveSessionIDWithHarness hardcoded branches on event.AgentID=='codex' (internal/hooks/runner.go:205) — if agent_id is already mutated before this check, downstream attribution is permanently wrong. Fix: trust raw_payload agent_id/agent_type for subagent events when is_subagent=true; do not let parent-session env override Claude-Code-supplied values. Files: internal/hooks/runner.go, internal/hooks/subagent_start.go, internal/agent/*.go. Add a regression test that asserts a subagent payload with agent_id='task-xxx' and agent_type='htmlgraph:researcher' produces exactly those values in the stored agent_events row.
