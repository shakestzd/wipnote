Description

When the orchestrator spawns a subagent via the Agent tool, the activity feed shows the Agent invocation as a leaf row with the subagent metadata (subagent_type, model, input/output tokens, api duration) but the tool calls the subagent itself made (Read/Bash/Grep/etc.) are not rendered as children of that Agent row. Causal lineage is broken at the dashboard layer. Expected: the Agent row should be expandable to show the subagent's session as a nested turn tree (its prompts, tool calls, results) so the user can drill from 'Research empty live_events feed' down into the Bash and Read calls the researcher actually ran. Actual: the Agent row only shows the spawn-call metadata; the subagent's tool calls either appear flat at top level under a separate session_id with no parent linkage, or are lost. Likely involves: pending_subagent_starts linkage, parent_span propagation in OTel, or the buildEventTreeOtel turn-anchor query not joining child sessions whose parent_session_id matches the spawning turn. Reproducer: spawn an htmlgraph:researcher agent that does several Reads and Bashes, watch the dashboard — the Agent row stays a leaf even though child activity exists in otel_signals/agent_events.
