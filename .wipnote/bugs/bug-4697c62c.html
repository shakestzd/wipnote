Description

Per-project serve-child runs two sql.DB writers concurrently against the same WAL SQLite file (the read DB handle plus the OTel indexer writer at serve_child.go:73). Combined with active hook ingest, every read query intermittently hits SQLITE_BUSY. The API handlers swallow the error and return a misleading success: api_tree.go:101 returns nil rows, treeHandler serializes [], the dashboard renders 0 events. api.go:331 calls QueryRow(...).Scan(&totalEvents) and discards the error, so /api/stats returns total_events:0. Same for active_sessions/cost_today/etc. Reproducer: open dashboard while hooks are firing; tree/stats endpoints intermittently return [] / zeros despite otel_signals having thousands of rows. Workaround discovered: kill _serve-child PID, parent supervisor respawns clean, reads succeed for a while until contention rebuilds. Two-part fix needed: (a) propagate Scan/Query errors instead of swallowing them so 500s are visible instead of false-success [] / 0, (b) eliminate the dual-writer contention (single writer connection, BEGIN IMMEDIATE serialization, or split read pool with WAL mode busy_timeout).
