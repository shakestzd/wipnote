During autonomous execution of plan-237fb251 via the execute skill on 2026-04-09, sub-agent dispatch hit a policy-level attribution guard that prevented sub-agents from starting features they did not create in their own session. Workaround: ran the dispatch directly in the orchestrator session, which was acceptable only because the 6 hardening slices formed a strictly linear blocked_by chain (no parallel dispatch needed). This is distinct from bug-d2d3fb3f (the SQLite contention bug fixed by feat-ab67561e / commit a1bca118). The per-(session_id, agent_id) attribution we landed addresses the DB-write race, but there is a separate, earlier guard somewhere in the execute skill or its feature-start call path that checks 'was this feature created in the current session' before allowing the claim. That check is what blocked the sub-agent path today. Investigation needs: (1) locate the guard — likely in plugin/skills/execute/SKILL.md or in the CLI's feature start permission logic; (2) decide whether the guard is still necessary now that (session_id, agent_id) attribution is the key; (3) if not, remove it so parallel dispatch works end-to-end. If still necessary, document when it fires and surface a clearer error message so the workaround (orchestrator-session execution) is discoverable. Priority: low. The workaround is fine for strictly-sequential chains like the hardening track. This only matters when a future track has independent parallel slices and we want multi-agent dispatch.
