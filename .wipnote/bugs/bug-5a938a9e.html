YOLO worktree teardown loses uncommitted .wipnote/ artifacts

Done High Priority

Part Of:

  • trk-08dcbb33

Description

When wipnote yolo executes a plan inside an isolated git worktree, the per-feature .wipnote/features/feat-*.html files created by the YOLO session are not staged or committed before the worktree is torn down. The HTML files are lost from main; only the code-change commits propagate. Concrete instance (2026-05-12): plan-2133adf2 was YOLO-executed, all 7 feature commits + 1 review-fix commit landed on main via merge f2e36d26c, but the 7 corresponding feat-*.html files survive only in .wipnote/.pre-merge-backup-20260512T040803Z/features/. wipnote feature show / lineage / trace all return work item not found for the IDs that git commits reference. Directly contradicts the causal-lineage value prop (defensibility-audit finding #1).
