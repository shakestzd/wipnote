Description

transcript_assistant.go writes assistant_text rows with parent_span = record.parentUuid. But parentUuid on assistant text records that follow tool calls points at the preceding tool_result record (type='user' but content is tool_result), not at the original user text prompt. Result: assistant_text.parent_span never equals any user_prompt.span_id in the DB, the primary client-side bridge match fails, and the dashboard falls back to a timestamp-window heuristic. Fix: in transcript_assistant.go, walk the parentUuid chain upward until we hit the nearest ancestor with type='user' AND content that is a text block (not a tool_result array). Use THAT ancestor's uuid as parent_span on the assistant_text row. Keep session_id/trace_id logic unchanged. Primary bridge match will then work directly without the timestamp fallback.
