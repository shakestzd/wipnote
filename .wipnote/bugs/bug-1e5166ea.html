Description

Regression from commit 388abbd3 (fix for bug-5d7220f3). When one orchestrator api_request spawns N parallel Task tool_uses in the same LLM turn, only the FIRST Task row absorbs the preceding api_request. Tasks #2..N have no _precedingApi and render without model pill or token/cost data — visually identical to 'OTel not firing'. Root cause: forward-scan exclusive absorption in _indexSpans (cmd/htmlgraph/dashboard/components/event-tree.js) + orchModel detection in 388abbd3 requires (api_request, non-Task tool) pair which may never exist if the orchestrator's turn consisted solely of Task dispatches. Fix direction: allow Task/Agent spans to share a reference to the nearest preceding orchestrator api_request (read-only share, not exclusive ownership), or second pass that inherits model from nearest preceding absorbed api_request. Test gap: api_otel_test.go exercises Go handlers, not the JS _indexSpans logic. Visible evidence: two HTMLGRAPH:SONNET-CODER Task rows with no model pill under an orchestrator that does show $3.41 / 4.22M TOK / 74 tools.
