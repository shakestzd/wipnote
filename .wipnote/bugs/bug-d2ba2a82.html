Hardcoded :4318 causes cross-project OTel data leakage when multiple projects are open

Description

roborev Finding 2: all Claude launches export to the same global OTLP endpoint 127.0.0.1:4318 but the receiver lives inside a per-project _serve-child. Only one child can bind that port. Sequence: Project A opens, A's child binds 4318. Project B opens, launcher's probePort sees 4318 'live' so ensureServeForOtel skips its own spawn, and B's env vars still point at 4318. Result: B's telemetry flows into A's otel_signals DB. Silent cross-project data mixing + misattribution. Fix: allocate a per-project OTLP port (hash of project_dir or registry-assigned), write it to .htmlgraph/.otlp-port, have both the launcher's buildClaudeLaunchEnv and the serve-child receiver read that file so they agree. Launcher env OTEL_EXPORTER_OTLP_ENDPOINT becomes http://127.0.0.1: instead of :4318. Keep :4318 as a default only when the project-specific file is absent.
