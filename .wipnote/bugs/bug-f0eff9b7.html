Description

Host landing page at http://localhost:PORT/ shows 1754 projects on a fresh devcontainer, with ~5531 directories under user cache dir. Almost all entries are anonymous (no slug) and named 001 or 002 with creation times clustered in the last ~80 minutes — i.e. they accumulate during normal session activity, not historically. Hypothesis: every htmlgraph CLI invocation from a different cwd auto-registers a new project entry keyed by path-hash (16 hex chars). With test runs, plugin build subprocesses, hooks firing from various cwds, and agent subprocesses, the registry fills with garbage. Repro: htmlgraph serve --port 8088 --bind 0.0.0.0; navigate to /; observe 1000+ entries. Real project (htmlgraph at path-hash 46c1de7d) appears once at top with timestamp matching serve start. Impact: host landing page is unusable; cache dir grows unbounded; project registry is no longer a reliable index. Investigation should identify: (a) the registration code path, (b) whether validation (real project root with .htmlgraph/ subdir) is missing, (c) whether sweep/cleanup commands handle this case, (d) safe cleanup strategy that preserves legitimate project entries.
