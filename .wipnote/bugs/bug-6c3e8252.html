Description

Discovered while investigating Bug A. SQLite has 9 sessions with status='active' from 2026-03-17 and 2026-03-18 (sess-* legacy ID format), e.g. sess-191b813d, sess-4d136dad, sess-73f58d23. These are old sessions whose JSONL files presumably no longer exist or are no longer in autoIngest's discovery scope, so the 60s sweep never re-checks them and they stay 'active' forever. They inflate the dashboard 'active sessions' stat. Fix: add a one-shot cleanup that flips status='active' to 'completed' for any session whose JSONL file is missing or whose mtime is older than (say) 24 hours. Run on serve startup and via 'htmlgraph cleanup stale-active'. Idempotent.
