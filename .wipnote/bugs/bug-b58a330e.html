Description

internal/otel/receiver/writer.go:46 constructs the SQLite DSN with '_pragma=journal_mode(WAL)' hardcoded, bypassing the fstype-aware BuildPragmas helper from commit 8cc7a775. On unsafe filesystems (virtiofs, FUSE, 9p, overlay) BuildPragmas correctly resolves to DELETE to avoid the WAL-SHM mmap SIGBUS, but NewWriter then re-opens the DB and forces WAL, persistently tipping the DB back to WAL mode. Downstream effect: hooks like user-prompt and subagent-start that re-apply DELETE on Open() fail because serve-child holds WAL connections (switching modes needs exclusive access). The hook binary swallows the open error and returns a no-op fallback, so no UserQuery or task_delegation events are written — activity feed goes silent, no error surfaces anywhere. Fix: drop the journal_mode clause from NewWriter's DSN and rely on BuildPragmas/ApplyPragmas. Keep other pragmas (busy_timeout, synchronous, foreign_keys) if they serve a single-writer purpose. Add a test that NewWriter on an unsafe-fs tmpdir does NOT flip journal_mode to WAL. Files: internal/otel/receiver/writer.go.
