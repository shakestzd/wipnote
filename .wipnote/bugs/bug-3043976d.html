Description

Follow-up to bug-6d235100. The first fix (mmap_size=0) eliminated the main-DB mmap crash but WAL mode still crashes in _walIndexAppend because the -shm (shared memory index) file is always mmap'd when journal_mode=WAL — that is not configurable via mmap_size. On virtiofs/fakeowner bind mounts (Docker Desktop macOS devcontainers), mmap is unsafe, so every write SIGBUSes and can corrupt the DB. Fix: at db.Open(), detect the backing filesystem for dbPath via unix.Statfs / /proc/self/mountinfo; choose journal_mode=WAL on safelisted native fstypes (ext4, xfs, btrfs, tmpfs, zfs) and journal_mode=DELETE on unsafe ones (fuse*, virtiofs, 9p, overlay upper layer, fakeowner). Default to DELETE when uncertain. Keep mmap_size=0 as the default. Expected files: internal/db/pragmas.go + new helper like internal/db/fstype_linux.go / fstype_other.go with build tags.
