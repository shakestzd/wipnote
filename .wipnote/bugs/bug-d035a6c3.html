The items aren't being worked on — they're frozen in-progress and only their HTML files get touched when state-sweep commits sync metadata. **Status diverges from actual development activity**: an item can sit in-progress for 15 days while git shows zero feature commits referencing it. Implication: any future stale-detection must cross-reference work-item status against **commit activity** (commits referencing the item ID, agent spawn events, session activity), not last-modified time on the HTML file — which sweep commits keep artificially fresh and would mask the staleness. ## Root cause — fix at the work-start layer (the orchestrator) The leak is at *work start*, not at *staleness detection*. Once an item is in-progress without a closing path, no downstream alert recovers it. The orchestrator must guarantee a finish for every start. Required behaviors: 1. **Start ⇒ finish contract** — every 'feature/bug/spike start' must have a paired completion path before session ends. The orchestrator owns this contract; agents are not trusted to self-complete. 2. **Post-agent verification, hard-enforced** — after a delegated agent returns, the orchestrator must query item status and call 'complete' if still in-progress. Currently this is a 'should' in the system prompt; it needs to be enforced (hook block / system-prompt assertion / automated post-agent CLI step). 3. **Session-exit reconciliation** — on session end (Stop hook), reconcile in-progress items started in this session: prompt the user (or auto-demote to todo) rather than silently leaving them in-progress. 4. **WIP limit enforcement** — currently reported, not enforced. Block 'feature start' past the WIP limit (or require an explicit override flag) so the orchestrator can't silently pile up new items on top of unfinished ones. 5. **Activity-aware staleness** — secondary safeguard. A 'htmlgraph stale' command using commit-activity heuristics catches items the primary fixes miss, but it is the backstop, not the primary fix. The primary fix lives in the orchestrator dispatch/return loop and the Stop hook. Stale-detection tooling is the safety net, not the cure.
