PROBLEM: wipnote plugin build-ports emits the same agent frontmatter fields to all three harness trees regardless of whether each harness honors them. Doc audit 2026-05-15 found timeout_mins emitted to Claude Code (silently ignored — not in its subagent schema) and flash/pro model shorthand emitted to Gemini (unverified vs documented full IDs). These are silent-failure bugs: caps/fields appear set but do nothing. APPROACH: bake a per-harness supported-frontmatter-fields allowlist into internal/pluginbuild so the generator strips (or errors on, behind a flag) any field a target does not support. Source of truth for allowed fields per harness should live near the manifest (packages/plugin-core/manifest.json) or as a typed table in internal/pluginbuild. ACCEPTANCE: (1) build-ports refuses to emit timeout_mins to Claude Code tree; (2) Gemini/Codex retain fields they support; (3) a unit test asserts the allowlist enforcement per harness; (4) unknown/unsupported field in a plugin/agents/*.md source produces a clear build-time warning naming the field+harness; (5) docs in plugin-development rules updated to describe the allowlist. FILES: internal/pluginbuild/ (generator + new allowlist table + tests), packages/plugin-core/manifest.json (if allowlist lives there), .claude/rules/plugin-development.md. RISK: must not break the currently-correct Gemini/Codex output; snapshot-test the generated trees before/after. RELATED: complements but does not replace the periodic upstream doc audit (audit catches NEW useful fields; this prevents silent misuse of the CURRENT contract).
