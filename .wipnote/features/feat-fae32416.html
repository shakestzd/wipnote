Pre-complete gate: block feature/bug/spike complete on uncommitted source

Todo Medium Priority

Part Of:

  • trk-0677c709

Description

Scope: add a Go CLI enforcement gate before feature complete, bug complete, and spike complete mark work done. Investigate command wiring under cmd/wipnote/ for the complete subcommands, and shared completion helpers if present. Also inspect the artifact commit path around commitWipnoteArtifact and commitPlanChange in internal/ so the new check runs before the .wipnote artifact-only commit path can make completion appear durable. Problem: completion currently commits only the .wipnote/*.html artifact and does not verify that tracked source changes are committed, so implementation files such as internal/pluginbuild/ can remain uncommitted after a feature is marked complete. Proposed behavior: before completing, run git status --porcelain semantics and detect tracked-file changes outside .wipnote/. Default behavior should be a hard block, not merely a warning, because completion is the durable done signal and standalone Codex/Gemini/single-agent usage lacks a manual orchestrator safety net. The refusal message should list offending relative file paths, explain that source changes must be committed first, and show exact remediation commands such as git add && git commit -m ": commit implementation"; provide an explicit --allow-dirty escape hatch for legitimate dirty-tree workflows, logging or printing that the bypass was used. Cross-harness requirement: this must live in the Go CLI completion path, not in a Claude-Code-only hook, so Claude Code, Codex, Gemini, and humans all receive identical enforcement. Acceptance criteria: 1. feature complete, bug complete, and spike complete refuse by default when git status --porcelain reports tracked modified/deleted/renamed/copied files outside .wipnote/. 2. The refusal output lists only relative offending paths and includes a clear git add plus git commit remediation example. 3. Dirty .wipnote artifacts alone do not block the normal completion artifact commit flow. 4. --allow-dirty bypasses the gate for feature/bug/spike complete and makes the bypass visible in command output. 5. Tests cover clean tree, dirty source tree, dirty .wipnote-only tree, and allow-dirty behavior. Likely files touched: cmd/wipnote/, internal/ artifact commit helpers near commitWipnoteArtifact and commitPlanChange, tests under cmd/wipnote/ or internal/ completion-related test files, docs/CLI help text if flags are documented there. Risks: false positives for legitimate dirty source worktrees; need to distinguish tracked source changes from generated .wipnote artifacts and possibly untracked files. Confirm interaction with the worktree/.wipnote gitignore exclusion documented in CLAUDE.md so the check does not accidentally block or ignore the wrong files.
