Description

Fix the persistence gap: `agent_events.teammate_name` and `team_name` columns exist (added by migrations in `internal/db/schema.go:123,128`) but `InsertEvent` never writes them. Result: SQL queries against these columns return NULL for every row. Add the corresponding fields to `AgentEvent` model, extend the INSERT statement, and verify hook handlers populate them on construction. **Implementation:** 1. Add fields to `AgentEvent` struct in `internal/models/event.go`: ```go TeammateName string `json:"teammate_name,omitempty"` TeamName string `json:"team_name,omitempty"` IdleReason string `json:"idle_reason,omitempty"` ``` 2. **Add a new ALTER TABLE migration in `internal/db/schema.go`** mirroring the existing teammate_name/team_name pattern at lines 123-132: `ALTER TABLE agent_events ADD COLUMN idle_reason TEXT` (idempotent, ignore "duplicate column name" error). Without this migration, slice 2's INSERT statement references a column that doesn't exist on un-migrated DBs and crashes at runtime. (Per opus HIGH-2 + sonnet HIGH-4 — same finding from both critics.) 3. Extend `InsertEvent` in `internal/db/event_repo.go`: add the three columns to the column list and parameter list. **Use `nullStr()`** (already imported in event_repo.go:52-59) to match the existing convention for the 9 other nullable text columns. Empty strings → SQL NULL on write; `sql.NullString.String` returns `""` for NULL on read; round-trip stays empty-string-clean for Go callers. (Per opus medium + q-empty-vs-null recommendation switch.) 4. **Update three specific SELECT read sites** (per opus HIGH-3 — explicit enumeration replaces vague "all consumers"): a. `GetEvent` in `internal/db/event_repo.go:81-124` — hand-rolled column-by-column SELECT; add the three columns to both the SELECT list and the Scan target list. b. `ListEventsBySession` in `internal/db/event_repo.go:132+` — narrow column subset; add the three columns. c. `ListEventsBySessionAsc` in `internal/db/event_repo.go:171+` — same as above. Plus: grep `cmd/htmlgraph/`, `internal/dashboard/`, `internal/sessionhtml/` for additional `SELECT … FROM agent_events` consumers; update each. Add a per-function round-trip test for the three columns in each. 5. **`TeammateIdle` handler refactor** (per sonnet HIGH-5 — the `recordSimpleEvent` shared helper trap): the current `TeammateIdle` handler at `internal/hooks/missing_events.go:113` delegates to `recordSimpleEvent`, which constructs a generic `AgentEvent` with no teammate fields. Adding fields to `AgentEvent` does NOT auto-propagate them through `recordSimpleEvent`. Two implementation options: a. **Inline the event construction** in `TeammateIdle` like `TaskCreated`/`TaskCompleted` do (preferred — clear, explicit, matches sibling handlers' patterns). b. Extend `recordSimpleEvent`'s signature to accept `teammateName`, `teamName`, `idleReason` strings. May cascade into other callers of `recordSimpleEvent` — verify call sites first. Pick (a) unless cascade risk in (b) is non-trivial. Document the choice in the implementer's PR. ## Why CLAUDE.md claims teammate identity is captured automatically. The research confirmed that step-prefix attribution in HTML works (via `addTaskStep`), but the DB-side claim is partially false — `agent_events.teammate_name` and `team_name` are NULL on every row because `InsertEvent` doesn't include them. Closing this gap turns the dashboard's per-teammate views (and any future analytics-by-teammate queries) from impossible-by-design to working. Without this fix, slice 1's auto-enablement would expand the surface of broken-by-omission behavior. Bundling the bug fix with the enablement plan ensures a coherent shipped behavior: when teams are auto-enabled, identity is also auto-persisted. ## Done When - `AgentEvent` struct has `TeammateName`, `TeamName`, `IdleReason` fields with `json:"...,omitempty"` tags - **New schema migration**: idempotent ALTER TABLE adds `idle_reason TEXT` to `agent_events` (mirroring teammate_name/team_name migrations at schema.go:123-132) - `InsertEvent` writes all three columns via `nullStr()` (NULL for empty strings, matching the existing convention for nullable text columns) - **Three SELECT read sites updated**: `GetEvent`, `ListEventsBySession`, `ListEventsBySessionAsc` all include and scan the three new columns; per-function round-trip tests pass - **`TeammateIdle` handler refactored** off `recordSimpleEvent` (or `recordSimpleEvent` extended): `event.TeammateName`, `event.TeamName`, `event.IdleReason` now propagate to the persisted `AgentEvent` - Insert + SELECT round-trip on a fresh DB: TaskCreated row with teammate_name=alice returns `alice` on `SELECT teammate_name FROM agent_events WHERE id=?` - Same round-trip for TeamName and IdleReason - Pre-existing rows with NULL teammate_name/team_name read correctly (no migration of historical data per design.non_goals) - Migration regression test: open a DB at the pre-slice-2 schema, run InsertEvent with new fields, verify no error and correct round-trip
