### What Changes Make Markdown rendering of `slice.what`, `slice.why`, and `slice.tests` explicit and reliable in the dashboard. The template already marks these fields with `data-markdown`; this slice hardens the behavior and updates styling so headings, lists, inline code, code fences, and paragraphs render cleanly inside cards. Keep `done_when` as a structured list of acceptance criteria instead of folding it into Markdown. It is both human-facing and machine-useful for compliance, TDD stubs, and execution prompts. ### Local Decision Prefer this YAML shape for rich card text: ```yaml what: | ### What Changes Markdown text... why: | ### Why This Matters Markdown text... done_when: - Concrete acceptance criterion - Another concrete acceptance criterion tests: | ### Unit - Test specific behavior ### Regression - Preserve existing behavior ``` ### Critic Revision Badge `Storage simplification` - replaced the proposed JSON migration with Markdown-native YAML fields. ## Why The text rendering problem can be solved without a storage migration. YAML literal blocks already preserve the formatting Markdown needs, and the renderer already has a Markdown hook. This is the fastest path to readable slice cards. ## Done When - `what`, `why`, and `tests` render as Markdown in slice cards. - Markdown rendering is sanitized and does not allow arbitrary script/HTML injection. - The plan skill instructs agents to use YAML literal blocks for Markdown-capable fields. - Existing plain-text YAML plans still render acceptably.
