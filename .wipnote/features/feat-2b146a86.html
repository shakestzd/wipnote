Description

Branch SessionStart output on CloudEvent.source so each entry-mode surfaces exactly the context that matters for THAT mode. Today we emit the same block regardless of whether the user started fresh, resumed, or just hit /clear. Proposed branches: SOURCE=startup, no active claim: Line 1: most-recently-touched in-progress item (likely pick-up candidate) Line 2: otel receiver status (from pass 1) SOURCE=startup, active claim exists: Line 1: 'active: — ' Line 2: otel receiver status SOURCE=resume: Line 1: active work item for THIS session (JOIN sessions + active_work_items ON session_id). Skip open-items list — the resumed transcript already has full context. Line 2: git-diff-stat since sessions.start_commit if something changed on disk during the gap. SOURCE=clear or SOURCE=compact: Line 1: active work item (just wiped context — this is the one case a FRESH reminder matters). Line 2: recent activity summary — last 3 tool calls from this session's agent_events since a synthetic 'context checkpoint'. Line 3: otel receiver status. Edge cases to resolve: - What if previous session's claim was completed by another agent between sessions? Verify claim still valid (status != done) before surfacing. - What if multiple sessions share the same active_work_items row via root sentinel? Dedupe. - Git-diff-stat should skip if untracked files only, otherwise it's noise. Depends on: pass 1 (the trim + autostart foundation). Do pass 1 first to establish the cleaner baseline, then layer source-awareness on top. Files: same as pass 1, plus CloudEvent parsing to read the source field.
