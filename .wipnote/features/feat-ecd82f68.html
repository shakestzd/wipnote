Description

The PreToolUse hook in `internal/hooks/pretooluse.go` checks whether the current session owns the active feature claim before allowing Write/Edit/MultiEdit. Today the check uses only the sub-agent's own session ID and refuses if `claim=none`. Sub-agents dispatched by an orchestrator that ran `htmlgraph feature start` see no claim under their own session and get blocked. **Implementation:** 1. **Locate the claim check.** Find the spot in `pretooluse.go` (or its callees) that resolves the claim by session ID -- likely a function like `resolveActiveFeatureClaim(sessionID)` or similar. 2. **Locate the parent-chain walker.** `internal/hooks/yolo_guard.go` already has `getSessionAndParent(sessionID)` (or similarly named) that walks the parent chain via `agent_events` (or `sessions`) table to find the orchestrator session. Reuse it; do NOT duplicate. 3. **Replace the lookup with a parent-chain walk.** When the current session has no claim, walk up via `getSessionAndParent` and check each ancestor session for an active claim. If any ancestor session owns a claim for the feature being touched, treat it as the effective claim and allow the operation. 4. **Logging:** when the claim is inherited via parent chain, log a structured note (e.g., `claim inherited: session= parent= feat=`) so the dispatch path is auditable. 5. **Conservative scope:** only walk the chain when `claim=none` for the current session. If the current session has its own (different) claim, leave existing behavior -- don't try to merge or transfer.
