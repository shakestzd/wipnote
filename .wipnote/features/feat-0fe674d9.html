Description

Remove the direct INSERT OR IGNORE into otel_signals from internal/hooks/transcript_user_prompt.go:191 and internal/hooks/transcript_assistant.go:298. Replace with NDJSON append to the session's events.ndjson — Verified by the feasibility critic that hooks have `sessionID` and `projectDir` in scope at the INSERT call sites (transcript_user_prompt.go:191 and transcript_assistant.go:298 receive both via the hook event payload and ResolveProjectDir helper). Implementation per q-write-coordination (recommended option-a): hooks call `ndjson.New(projectDir, sessionID).WriteBatch(...)`. The per-session collector's indexer picks them up on next tick. Hidden work flagged by the feasibility critic: the current INSERT writes raw column values (signal_id, span_id, ts_micros, etc.); the NDJSON line format is a full `otel.UnifiedSignal` struct. The mapping from raw transcript record fields to UnifiedSignal must be worked out explicitly during implementation — this is not a one-line replacement. ## Why Eliminates the dual-writer contention that started this whole story (bug-4697c62c). Hooks become NDJSON producers, indexer becomes the sole SQLite writer. Without this slice, the indexer goroutine and the hook processes will fight for the canonical DB write lock under load. ## Done When - grep -rn 'INSERT.*otel_signals' internal/hooks/ returns zero matches - A transcript hook invocation in a fresh session produces a row in otel_signals within one indexer tick after the hook returns - Hook latency under load (1k invocations) regresses by less than 5% versus the direct-SQLite baseline
