### What Changes Add APIs and CLI commands for updating one slice at a time: - `plan approve-slice` - `plan reject-slice` - `plan answer-slice-question` - `plan set-slice-status` Store live review state in SQLite during interaction, then reflect durable lifecycle state in the YAML plan document on explicit save/promote operations. Lifecycle states should distinguish plan state from slice state: - plan: `draft | active | completed` - slice approval: `pending | approved | rejected | changes_requested` - slice execution: `not_started | promoted | in_progress | done | blocked | superseded` ### Local Decision Plan-level finalization should become "complete plan" and should not be required before approved slices can move. ### Critic Revision Badge `All-or-nothing gate removed` ### Section-Naming Contract All CLI mutation commands (approve-slice, reject-slice, answer-slice-question, set-slice-status) MUST use the section-naming contract defined in slice-1: section='slice-' for slice-level state, section='slice--question-' for question answers. Existing IsPlanFullyApproved at internal/db/plan_feedback.go:71 must continue to pass after the changes — add a regression test before introducing slice-local sections. ## Why The current finalization gate turns review into a batch process. Large plans need incremental approval: a user should be able to approve and implement slice 1 while slice 4 still has unanswered questions. ## Done When - A single slice can be approved/rejected without changing unrelated slices. - Slice-local question answers persist and are returned by plan feedback APIs. - The plan can remain active while some slices are approved, rejected, pending, or done. - Existing `plan_feedback` consumers continue working for legacy YAML plans. - Section-naming convention for plan_feedback documented in slice-1 and consumed by slice-4 — slice-local approvals use section='slice-', slice-local question answers use section='slice--question-' - validSectionRe at cmd/htmlgraph/api_plans.go:286 extended to accept `slice-\d+-question-[a-z0-9-]+` so the answer-slice-question command is not blocked at the HTTP gate. The existing normalization at api_plans.go:302-306 (underscore-to-hyphen) is mirrored for the new pattern. Regression test added: an HTTP POST to /api/plans//feedback with section='slice-3-question-q-foo' returns 200, not 400.
