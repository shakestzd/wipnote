Description

Add the launch planning and guard machinery needed for mutating launchers to run in managed worktrees, but do not flip host production defaults unconditionally in this slice. For devcontainer/local-dev profiles the plan may default to managed worktrees behind an explicit config gate; for host production profiles it starts as warn-only until doctor/migration and session-family hardening have landed. Note: `wipnote yolo` already implements worktree-per-track/feature with `--no-worktree` (`cmd/wipnote/yolo.go`). This slice generalizes that model to claude/codex/gemini and adds the dirty-main guard — it does not build worktree plumbing from scratch. Reuse existing `internal/worktree` and command-level worktree helpers. Add explicit flags: - `--worktree` / `--no-worktree` - `--in-place` as an alias for intentional local checkout mutation - `--base ` for advanced users - `--work-item ` as a harness-neutral spelling for feature/bug/track Add a dirty-main guard that explains the managed-worktree recommendation and refuses mutating in-place launch only when the configured rollout mode says enforcement is enabled. The mode contract must record whether the launch used isolation, warn-only, or explicit in-place override.
