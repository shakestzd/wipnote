Description

On bug/feature create (cmd/wipnote/workitem_create.go), run normalized title+description similarity against open and recently-closed items (query via internal/db/feature_repo.go). On a strong match, auto-attach a relates_to edge and a needs-triage-dup tag that the dashboard and wipnote check surface; when a TTY is present, show an interactive confirm prompt first. Similarity is stdlib-only (normalized token/trigram), tuned to a documented threshold over a documented, configurable similarity window (default ~30d) with a recorded rationale. Dedup must degrade to a graceful no-op (not an error) when the read index is empty/absent (fresh clone / CI), so create never fails because the index has not been built.

## Problem

## Decisions
**Trigger:** bug/feature create whose title+description is near-identical
to an open or recently-closed item within the configurable similarity
window.
**State:** durable - query existing items from the read index/artifacts;
decision is content-based, not session-based; graceful no-op when the
index is empty/absent.
**Payload:** auto-attach relates_to edge + needs-triage-dup tag;
interactive confirm only when a TTY is present; surfaced in dashboard
and wipnote check. The similarity window is a documented configurable
flag (default ~30d) and the threshold has a recorded rationale.
**Rejected:** a new fuzzy-match runtime dependency (rejected - stdlib
normalized token/trigram is sufficient and the constraint forbids new
deps); failing create when the index is empty (rejected — fresh
clone/CI has no index; dedup must no-op gracefully); relying on agents
to manually search before creating (rejected - advisory-only is what
failed; detection belongs in the binary).
