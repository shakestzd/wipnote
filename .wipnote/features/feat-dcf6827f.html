Description

Fixes bug-71fc095f. In internal/hooks/session_start.go SessionStart path, the current resolution prefers ResolveProjectDir's CWD walk-up. This causes a split-brain when the user cd's between projects during a single Claude Code session: each cd writes the session HTML to a different project's .htmlgraph/sessions/, and each project's reindex then pulls those events into its own SQLite, creating phantom session rows in the wrong project's DB. Fix: after computing projectDir via paths.ResolveProjectDir, check if CLAUDE_PROJECT_DIR env var is set AND points at a directory containing .htmlgraph/. If yes, use CLAUDE_PROJECT_DIR instead — it's set by Claude Code at session launch and stays stable across cd inside the session. CWD-based walk-up becomes the fallback only when CLAUDE_PROJECT_DIR is unset (e.g., direct htmlgraph CLI invocation outside a Claude Code session). Guardrail: only prefer CLAUDE_PROJECT_DIR when HTMLGRAPH_SESSION_ID is also set in the same env, which indicates the env was populated by a current Claude Code process rather than inherited from a parent shell (could be stale across sessions). Investigation evidence (2026-04-11): - session 8d53982f exists as a 104 KB HTML in shakestzd/.htmlgraph/sessions/ AND a 167 KB HTML in htmlgraph/.htmlgraph/sessions/ - same session_id, both files physically on disk, both reindex targets - shakestzd DB has 451 tool_call events all with source=reindex (parsed from the stray HTML) - htmlgraph DB has 428 reindex + 1319 hook + 71 ingest events for the same session_id - root cause: CWD at hook-fire time was shakestzd during doorway testing - existing data is historical and accurate at moment of write — the fix is forward-looking only, do not revise history Do NOT delete or move the existing stray HTML file. Leave historical data as-is. The fix prevents future occurrences only.
