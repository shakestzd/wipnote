Description

The /api/graph endpoint currently takes ~7.2s to respond for 505 sessions + 436 features + 176 edges + 43,597 agent_events on the htmlgraph project, and the dashboard waits for the full payload before rendering. Users see a multi-second blank screen followed by a D3 force simulation bouncing nodes into place from random positions. Scope (three stacked fixes): 1. Query optimization (Approach A — biggest lever). Profile the five queries in cmd/htmlgraph/api_graph.go. Primary suspect: the sessions-with-activity filter at lines 196-209 uses correlated EXISTS + GROUP BY HAVING COUNT(*) > 5 over 43k events. Replace with a single aggregated query. Add indexes on agent_events(session_id, feature_id) if missing. Target: /api/graph responds in under 500ms. 2. Bounce fix (Approach C — cheap win). Seed D3 node positions from a stable hash of the node ID before running the force simulation, and start the simulation at alpha(0.3) instead of alpha(1). Nodes appear in roughly stable positions and settle with a small shiver instead of bouncing from random. 3. Optional NDJSON streaming (Approach B) — only if (1) and (2) together still feel slow. Stream nodes and edges as chunked NDJSON lines so the client can render incrementally. Deferred unless needed. Out of scope: adding new graph features, changing the visual design, or touching the SSE stream infrastructure.
