Remove receiver.New/Start calls from cmd/htmlgraph/serve_child.go (lines 99-121 region). The embedded OTLP endpoint on the project-hashed port is deleted. Update the dependent sites identified by feasibility audit: claude_serve_autostart.go:51,141 (ensureServeForOtel + MaybeShowOtelNotice LoadConfigFromEnv calls), claude_env.go:96 (otelEndpointFromEnv), and internal/otel/receiver/port.go (deleted outright). Rename ensureServeForOtel to ensureServeForDashboard with revised semantics: serve is no longer auto-spawned for telemetry reasons. It is only auto-spawned when (a) a CLI command requires semantic-op execution or (b) the user explicitly opens the dashboard. Keep the PID-lockfile and TCP-probe pattern (checkServeLock, writeServeLock, spawnDetachedServe) intact for the dashboard path. Rework MaybeShowOtelNotice to display a generic telemetry-status message (derived from the presence/absence of active .collector-pid files in .htmlgraph/sessions/) rather than referencing a specific endpoint. Update receiver tests (writer_test.go, receiver_test.go) to construct receivers in-process with NDJSONSink rather than via the old serve mount. ## Why This is the slice that actually shuts the contention class down: once the OTLP receiver is gone from serve and collectors are the only ingest path, bug-08681166 (busy_timeout lost) stops reaching OTel writes entirely. The rename clarifies serve role as a reader plus semantic-op singleton, not a telemetry endpoint. Deleting port.go eliminates the multi-project-collision history documented in bug-d2ba2a82. Directly supports goals Eliminate OTel writer collisions and Lifecycle independence from serve. The MaybeShowOtelNotice rework is necessary because the static port derivation that notice referenced no longer exists — the user-facing message must instead describe per-session collectors. ## Done When - htmlgraph serve no longer binds any OTLP port; scanning running processes for listeners in range 4318-5317 shows only otel-collect children, never serve. - ensureServeForDashboard auto-spawns serve only when a semantic op or dashboard request requires it; launching htmlgraph claude with the dashboard already running does NOT re-probe any OTLP port. - internal/otel/receiver/port.go is deleted; no import references remain in the codebase (verified via go build). - claude_env.go:96 otelEndpointFromEnv is removed or reworked to read from active collector-pid files rather than a project-hashed port. - MaybeShowOtelNotice shows a message that does not reference a fixed port; instead it describes per-session collector behavior. - The specific SQLITE_BUSY cascade symptom of bug-08681166 (OTel writer colliding with hook writers) cannot reproduce because no OTel writer lives inside serve.
